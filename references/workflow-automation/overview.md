# Workflow Automation — DLP incident management and automated remediation

## What it is

Workflow Automation is a Zscaler application that enables governance admins to automate the management and resolution of Data Loss Prevention (DLP) incidents generated by ZIA (Internet & SaaS). It is not a general-purpose workflow tool — it is specifically scoped to **ZIA DLP incidents** (Tier A — vendor/zscaler-help/what-workflow-automation.md).

It provides a closed-loop incident management interface where admins can review, remediate, escalate, and close DLP incidents, with configurable automated workflows to handle repetitive incident response at scale.

## Core function

Workflow Automation ingests ZIA Data Protection incidents (generated from DLP policies defined in ZIA) and provides:

1. **Centralized incident view**: All DLP incidents in one place, with full metadata including the data that triggered the incident
2. **Incident grouping and prioritization**: Group incidents into groups; assign priorities; assign to different admins
3. **Manual remediation actions**: Investigate, notify user, escalate, create ticket, close incident
4. **Automated workflows**: Trigger configurable sequences of actions automatically when an incident matches defined criteria

## Workflow concepts

### Workflow templates (predefined)

Workflow Automation ships with 9 predefined workflow templates:

| Template | What it does |
|---|---|
| Auto Close Data Protection Incident With Resolution Label | Sets status to Resolved and adds a resolution label |
| Auto Close Data Protection Incident | Sets status to Resolved |
| Auto Create Tickets | Creates a ticket in the configured ticketing integration (ServiceNow or Jira) |
| Auto Escalate | Escalates to user's manager or an approver if manager not found |
| Auto Notify | Notifies the user who generated the incident via configured channel (email, Slack, Teams) |
| Auto Notify User and Close Incident | Notifies user; closes incident if no response within configurable time period (seconds) |
| Auto Notify User and Concurrently Escalate | Notifies user and simultaneously escalates to manager/approver — no wait for user response |
| Auto Notify User and Escalate | Notifies user; escalates to manager/approver if user doesn't respond within configurable time |
| Auto Notify User and Escalate to Manager | Notifies user; optionally escalates to manager if no response |

### Custom workflows

Admins can also build custom workflows by choosing and configuring individual steps and actions without using a predefined template.

### Workflow mappings

After creating a workflow, admins specify which incidents trigger it by mapping the workflow to incident attributes. When an incident arrives that matches those attributes, the workflow fires automatically.

### Notification channels

End-user notifications can be delivered via:
- Email
- Slack
- Microsoft Teams

### Escalation chain

Escalation targets: the user's manager (from directory), or a designated approver. If the manager is not found in the system, escalation goes to the approver instead.

## Incident management concepts

| Concept | Description |
|---|---|
| **Incidents page** | Central list of all DLP incidents with metadata and status |
| **Incident Details page** | Per-incident view including the data that triggered the incident |
| **Incident Groups** | Group individual incidents for bulk handling; assign groups to specific admins |
| **Priorities** | Configurable priority levels for incident groups |
| **Incident Group Mappings** | Rules that automatically assign incidents to groups based on attributes |
| **Approvers** | Designated escalation recipients (beyond manager chain) |
| **Duplicate Incidents** | Detection and handling of repeated incidents from the same activity |

## Integrations

### Ticketing systems

- **ServiceNow**: Native integration for ticket creation
- **Jira Software**: Native integration for ticket creation

These are used by the "Auto Create Tickets" workflow template.

### Identity / user directory

Workflow Automation relies on user directory data to resolve manager relationships for escalation. User attributes are managed within the product.

### ZIA

Workflow Automation is dependent on ZIA — it consumes ZIA DLP incident data. No ZIA subscription means no Workflow Automation incidents. The integration is read-only from Workflow Automation's perspective (it does not modify ZIA policies).

## Roles and permissions

Workflow Automation has its own RBAC:
- **Governance admins**: Manage workflows, configure mappings, view all incidents
- **Approvers**: Designated recipients for escalated incidents; can approve or reject
- **Integration users**: System accounts for ticketing system integration
- **User roles**: Regular users who receive notifications and can respond/justify

Roles and permissions are managed within Workflow Automation (not inherited from ZIA admin roles).

## Audit logs

The product has its own audit log surface (`About Audit Logs` documented in help portal). Actions taken within Workflow Automation — workflow changes, incident state changes, user notifications — are auditable.

## Key operational notes

- Workflow Automation's scope is **ZIA DLP only**. It does not handle ZPA logs, ZIA threat events, or any other Zscaler product's alerts. Operators asking about automating response to ZIA web filtering blocks, ZIA sandbox detections, or ZPA access events should look elsewhere.
- The time period for user response in templates like "Auto Notify User and Close Incident" is configurable in **seconds**, not minutes or hours. This is an unusual unit — worth flagging when operators configure response windows.
- Concurrent escalation (the "Auto Notify User and Concurrently Escalate" template) is distinct from sequential escalation — it escalates immediately in parallel with the user notification, not after waiting for user response. This is useful for high-severity incidents where manager awareness is required regardless of user response.
- User notifications leverage the ZIA-connected user identity to identify the user who triggered the incident. If a user lacks a manager record in the directory, escalation falls back to the configured approver.

## What Workflow Automation is not

- Not a SOAR platform. It is DLP-incident-specific, not a general security orchestration product.
- Not a ZPA workflow tool. ZPA has its own policy management; this product is ZIA DLP only.
- Not an email security tool. Slack/Teams/email notifications are delivery channels for DLP incident notifications, not inbound email security.
- Not a SIEM integration layer. For log forwarding to SIEM, use ZIA NSS (Network Security Services) or log streaming.
