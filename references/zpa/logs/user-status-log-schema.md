# ZPA User Status log schema (LSS User Status log fields)

Authoritative field-level reference for ZPA User Status logs — the LSS log type that records **ZCC authentication state, device posture evaluation, and session enrollment** events. Derived directly from Zscaler's *Understanding User Status Log Fields* article (vendored PDF).

## Purpose and scope

User Status logs are generated by LSS when a ZCC client connects to or disconnects from ZPA. Each record captures:

- Whether the user successfully authenticated to ZPA (SessionStatus)
- Which posture profiles passed or failed on the device
- Which trusted networks ZCC detected at authentication time
- ZCC version and client type
- SAML attributes delivered by the IdP

**User Status vs User Activity**: these are distinct LSS log types with different records and different diagnostic value.

| | User Status | User Activity |
|---|---|---|
| When generated | At ZCC authentication / session change | Per ZPA application connection |
| Primary use | Diagnose ZCC enrollment failures, posture failures, auth failures | Diagnose access denials, latency, connector selection |
| Granularity | One record per session state change | One record per application connection |
| Schema doc | This file | [`./access-log-schema.md`](./access-log-schema.md) |

User Status logs answer "why can't this user connect to ZPA at all?" User Activity logs answer "why can't this user reach application X through ZPA?"

## Field inventory

Per *Understanding User Status Log Fields*, pp.1–7. All field names are **case-sensitive** per *Understanding the Log Stream Content Format*, p.3.

### Session identity

| Field | Description | Format specifiers |
|---|---|---|
| `LogTimestamp` | Timestamp when the log was generated | `s`, `j`, `J` |
| `Customer` | The customer name | `s`, `j`, `J`, `o` |
| `Username` | The user name (email address in most IdP configurations) | `s`, `j`, `J`, `o` |
| `SessionID` | The TLS session ID | `s`, `j`, `J`, `o` |

### Session status

| Field | Description | Values | Format specifiers |
|---|---|---|---|
| `SessionStatus` | The status of the session | `ZPN_STATUS_AUTHENTICATED` — user successfully authenticated; `ZPN_STATUS_AUTH_FAILED` — authentication failed; `ZPN_STATUS_DISCONNECTED` — user disconnected from a Public or Private Service Edge | `s`, `j`, `J`, `o` |

### Client version and type

| Field | Description | Format specifiers |
|---|---|---|
| `Version` | The Zscaler Client Connector version | `s`, `j`, `J`, `o` |
| `ClientType` | Client type for the request: Zscaler Client Connector, ZPA LSS, or Web Browser | `s`, `j`, `J`, `o` |

### Network location

| Field | Description | Format specifiers |
|---|---|---|
| `ZEN` | The Public Service Edge selected for the connection | `s`, `j`, `J`, `o` |
| `CertificateCN` | The certificate common name | `s`, `j`, `J`, `o` |
| `PrivateIP` | The private IP address of Zscaler Client Connector | `s`, `j`, `J`, `o` |
| `PublicIP` | The public IP address of Zscaler Client Connector | `s`, `j`, `J`, `o` |
| `Latitude` | Latitude coordinate of the ZCC location | `f`, `o` |
| `Longitude` | Longitude coordinate of the ZCC location | `f`, `o` |
| `CountryCode` | Country code of the ZCC location | `s`, `j`, `J`, `o` |
| `City` | City of the client | `s`, `j` |
| `ZENLatitude` | Latitude coordinates of the Public Service Edge | `f`, `o` |
| `ZENLongitude` | Longitude coordinates of the Public Service Edge | `f`, `o` |
| `ZENCountryCode` | Country code of the Public Service Edge | `s`, `j`, `J`, `o` |

### Timestamps

| Field | Description | Format specifiers |
|---|---|---|
| `TimestampAuthentication` | Timestamp (microseconds) when ZCC was authenticated | `s`, `j`, `J` |
| `TimestampUnAuthentication` | Timestamp (microseconds) when ZCC was unauthenticated | `s`, `j`, `J` |

### Byte counters

| Field | Description | Format specifiers |
|---|---|---|
| `TotalBytesRx` | Total bytes received during the session | `d`, `x`, `f`, `o` |
| `TotalBytesTx` | Total bytes transmitted during the session | `d`, `x`, `f`, `o` |

### Device and identity

| Field | Description | Format specifiers |
|---|---|---|
| `Idp` | Name of the identity provider as configured in the Zscaler Admin Console | `s`, `j`, `J`, `o` |
| `Hostname` | Name of the device as reported by ZCC | `s`, `j`, `J`, `o` |
| `Platform` | Platform on the device as reported by ZCC (e.g., `windows`, `macos`) | `s`, `j`, `J`, `o` |

### Posture and trusted networks

| Field | Description | Format specifiers |
|---|---|---|
| `TrustedNetworks` | Unique IDs of trusted networks ZCC determined for this device | `s`, `j`, `J`, `o` |
| `TrustedNetworksNames` | Names of trusted networks ZCC determined for this device | `s`, `j`, `J`, `o` |
| `SAMLAttributes` | List of SAML attributes reported by the IdP (colon-separated key:value pairs, comma-delimited) | `s`, `j`, `J` |
| `PosturesHit` | Posture profiles ZCC successfully verified for this device (comma-delimited) | `s`, `j`, `J`, `o` |
| `PosturesMiss` | Posture profiles ZCC failed to verify for this device (comma-delimited) | `s`, `j`, `J`, `o` |

### Client-to-client FQDN registration

These fields are relevant only when ZPA client-to-client communication is configured.

| Field | Description | Values | Format specifiers |
|---|---|---|---|
| `FQDNRegistered` | Status of the hostname for the client-to-client connection | `true` or `false` | `s` |
| `FQDNRegisteredError` | Error status of the registered hostname | `FQDN_MATCH` (hostname matches regex), `FQDN_NO_MATCH`, `FQDN_BAD_SCHEMA` (c2c disabled for PSE), `FQDN_BAD_REGEX` (invalid regex in admin console), `FQDN_TOO_MANY_REGEX` (multiple regex configured), `FQDN_NO_ENTRIES` (no regex configured), `FQDN_EMPTY` (FQDN from ZCC is null/empty), `BROKER_NOT_ENABLED` (c2c disabled for PSE), `CUSTOMER_NOT_ENABLED` (c2c disabled for customer) | `s` |

### Microtenant

| Field | Description | Format specifiers |
|---|---|---|
| `MicroTenantID` | Microtenant ID of the user accessing the application | `s`, `j` |

## Joining User Status with User Activity logs

The shared key between User Status logs and User Activity logs is **`Username`** — in both log types this carries the user's email address as enrolled in the IdP.

Join pattern:

```spl
index=$INDEX_ZPA_STATUS Username=$USER earliest=-24h
| fields Username SessionStatus PosturesHit PosturesMiss Version Hostname Platform
| rename SessionStatus as status_event
| join type=left Username
    [search index=$INDEX_ZPA Username=$USER earliest=-24h
     | stats count values(Application) as apps_accessed values(ConnectionStatus) as conn_states
       by Username]
| table Username status_event apps_accessed conn_states PosturesHit PosturesMiss Version Hostname
```

Interpretation guide:
- `SessionStatus=ZPN_STATUS_AUTH_FAILED` with no matching User Activity records = user never established a ZPA session; diagnose IdP/SAML configuration.
- `SessionStatus=ZPN_STATUS_AUTHENTICATED` with no User Activity records = user enrolled but accessed no applications; check App Segment configuration or access policy.
- `PosturesMiss` non-empty + access denials in User Activity = posture policy is blocking application access.

**Index note**: User Status logs land in a separate LSS receiver from User Activity logs by default — they may be in a different Splunk index. The index variable `$INDEX_ZPA_STATUS` is used here to distinguish it from `$INDEX_ZPA` (User Activity). Confirm your LSS receiver configuration to verify whether they are co-indexed.

## Splunk: diagnosing ZCC connection failures

### `user-status-auth-failures`

**Purpose:** Find users with repeated ZPA authentication failures in the last 7 days — first-pass triage for "user can't connect to ZPA" reports.

```spl
index=$INDEX_ZPA_STATUS SessionStatus=ZPN_STATUS_AUTH_FAILED earliest=-7d
| stats
    count as failure_count
    values(Hostname) as hostnames
    values(Platform) as platforms
    values(Version) as zcc_versions
    values(PublicIP) as source_ips
    dc(SessionID) as unique_sessions
    by Username
| sort -failure_count
| rename failure_count as auth_failures
```

Notes:
- High `auth_failures` with consistent `zcc_versions` points to a version-specific regression.
- High `auth_failures` with varying `source_ips` suggests roaming users hitting geo-specific auth issues.
- `ZPN_STATUS_AUTH_FAILED` in User Status does not always surface in ZIA web logs — ZCC may not be forwarding traffic at all if ZPA auth fails.

### `user-status-posture-miss-analysis`

**Purpose:** Identify posture profiles that are failing most frequently across the fleet — useful for diagnosing sudden access failures after a posture policy update.

```spl
index=$INDEX_ZPA_STATUS PosturesMiss!="" earliest=-24h
| mvexpand PosturesMiss
| stats
    count as miss_count
    dc(Username) as affected_users
    values(Platform) as platforms
    by PosturesMiss
| sort -miss_count
```

Cross-check `PosturesMiss` values against the policy configured under **Policy > Access Control > Device Posture** in the ZPA admin portal to confirm which specific posture checks are generating failures.
