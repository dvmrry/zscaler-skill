# ZCC End-User Notifications — block pages, alerts, and in-app messages

## 1. Taxonomy — three distinct notification surfaces

Operators routinely conflate three separate notification mechanisms in the Zscaler stack. The mechanisms live in different places, are configured differently, and target different user experiences. Mixing them up produces incorrect troubleshooting paths.

### 1.1 ZCC-emitted notifications

Native OS notifications (toasts on Windows, alerts in macOS Notification Center, system notifications on Android) generated by the Zscaler Client Connector agent process running on the device. These originate from ZCC itself, not from the ZIA or ZPA cloud. Examples:

- Tunnel established / tunnel disconnected
- Captive portal detected
- ZPA reauthentication required
- Device posture check failure
- AUP prompt (in-app overlay)
- Service Disaster Recovery mode detected
- App update available

ZCC-emitted notifications are configured under **Administration > Client Connector Notifications > End User Notifications** in the ZCC Portal. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`]

### 1.2 In-app messages

Text and status codes displayed inside the ZCC tray/menu UI — for example, "Connected" / "Disabled" / "No Service" labels, error code numbers, and per-service status rows (ZIA row, ZPA row, ZDX row). These are not OS-level pop-ups; they appear only when the user opens the ZCC interface. In-app message text is largely determined by ZCC client version and is not directly configurable by admins in the notification settings UI.

### 1.3 ZIA-emitted block pages

HTML pages served by a ZIA Public Service Edge when a URL filtering or Cloud App Control rule blocks a request. These pages are rendered in the user's browser, not by ZCC, and are not the same as ZCC notifications. The user's browser receives an HTTP 403 response with an HTML body; ZCC is merely the tunnel through which the blocked request traveled. Block page content (logo, text, support links) is configured in ZIA, not in the ZCC Portal.

For ZIA block page configuration and how URL filtering rules trigger them, see [`../zia/url-filtering.md`](../zia/url-filtering.md).

## 2. What admins can configure

Configuration lives at **Administration > Client Connector Notifications** in the ZCC Portal. Two tabs exist: **End User Notifications** and **Acceptable Use Policy (AUP) Settings**. A third path — **Notification Templates** — replaces the End User Notifications tab when templates are in use; when templates are active, the standard tab does not appear. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`]

### 2.1 End User Notifications tab

The following toggles are available [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`]:

| Setting | Description | User-overridable |
|---|---|---|
| Enable Notifications by Default | Master toggle; enabled at enrollment time. | Yes |
| Enable App Updates Notifications | Notifies users when a ZCC upgrade is available. | No |
| Enable Service Status Notifications | Notifies users when a Zscaler service is in Disaster Recovery (DR) mode. | No |
| Enable ZIA Notifications | Enables ZIA-sourced notifications delivered through ZCC — for example, DLP notifications. | No |
| Enable Notifications for ZPA Reauthentication | Prompts users for ZPA re-authentication when required. | Yes |
| Show ZPA Reauthentication Notifications Every (In Minutes) | Interval in minutes for repeating ZPA reauth prompts (range: 2–1440). Enabled by default. | No |

### 2.2 Zscaler Notification Framework (Windows and macOS only)

Requires ZCC 4.2 or later on both Windows and macOS. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`]

| Setting | Description |
|---|---|
| Custom Timer (In Seconds) | How long a non-persistent notification stays on screen (range: 5–60 seconds). |
| Enable Persistent Notifications | Overrides Custom Timer. Critical notifications (ZPA reauthentication, captive portal detection, system reboot request) remain on screen until dismissed. |
| ZIA Notification Persistent | Overrides Custom Timer for ZIA notifications. Currently applies only to DLP notifications. |

### 2.3 Acceptable Use Policy (AUP) Settings tab

AUP is a full-screen in-app overlay that ZCC displays before allowing users to connect. It is not a web page block; it is rendered by ZCC itself. [Source: `vendor/zscaler-help/configuring-acceptable-use-policy-zscaler-app.md`]

| Setting | Options |
|---|---|
| AUP Frequency | Never / After User Enrollment / Daily / Weekly / Custom (1–180 days) |
| AUP Message | Free-form HTML with static tags; images must be internet-accessible URLs |
| Preview AUP Message | Admin-side preview of how users see the AUP |

Configuring branding (logo, corporate colors) within the AUP message body is possible through the HTML content field. Multi-language support within AUP is not described in the vendor source for this tab; see Section 6.

### 2.4 Notification Templates tab

When notification templates are enabled, this tab replaces the End User Notifications tab. Templates allow per-notification-type customization of text beyond the global toggles. The help portal references a separate article ("Configuring Notification Templates for Zscaler Client Connector") that was not captured in available vendor sources. Details on template fields and branding options are not confirmed from reviewed sources; see [Deferred — ZCC end-user notifications](#deferred-items).

## 3. Notification triggers

The following events surface ZCC-emitted notifications when the corresponding toggle is enabled. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`; `references/zcc/forwarding-profile.md`]

| Event | Notification type | Persistent (with framework) |
|---|---|---|
| ZCC app update available | OS toast / tray alert | No |
| ZIA service in Disaster Recovery mode | OS toast | No |
| ZIA DLP notification (policy-triggered) | OS toast | Yes (when ZIA Notification Persistent enabled) |
| ZPA reauthentication required | OS toast or tray prompt | Yes (when Persistent Notifications enabled) |
| Captive portal detected | OS toast | Yes (when Persistent Notifications enabled) |
| System reboot required | OS toast | Yes (when Persistent Notifications enabled) |
| Tunnel up / service enabled | In-app status change | N/A |
| Tunnel down / service disconnected | In-app status change | N/A |
| ZIA sign-in failure | In-app error message | N/A |
| Trusted Network state change | In-app status change (trusted / untrusted) | N/A |
| Certificate trust failure | In-app error or OS-level warning (platform-dependent) | N/A (not confirmed as ZCC-emitted toast) |
| Device posture check failure | In-app status / error | N/A (specific notification toast for posture failure not confirmed; see Deferred) |

Notes on specific triggers:

- **Captive portal.** ZCC detects captive portals via HTTP probe interception. Detection triggers a persistent notification (when the framework is enabled) so users know to open the portal in a browser. The captive portal grace period (`captivePortalWebSecDisableMinutes`) is configured on the App Profile (newer tenants) or the global FailOpenPolicy object; see [`./forwarding-profile.md § Fail-open policy`](./forwarding-profile.md). The notification and the grace-period timer are related but independently configured.
- **ZPA reauthentication.** The interval between repeated prompts is set in minutes (2–1440). If a user dismisses without authenticating, the prompt recurs at the configured interval. The per-user toggle allows users to disable these prompts from within ZCC.
- **ZIA DLP notifications.** These are forwarded from the ZIA cloud through the ZCC agent; they are not generated by ZCC from local policy evaluation. Enable ZIA Notifications must be on.

## 4. Per-platform support matrix

The notification subsystem behaviors differ across platforms. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`; `references/zcc/install-parameters.md`]

| Feature | Windows | macOS | Linux | iOS | Android | ChromeOS |
|---|---|---|---|---|---|---|
| ZCC-emitted OS toasts | Yes (Windows Toast API) | Yes (macOS Notification Center) | Not confirmed | Not confirmed | Yes (Android system notification) | Not confirmed |
| Zscaler Notification Framework | Yes (v4.2+) | Yes (v4.2+) | No | No | No | No |
| Persistent Notifications | Yes (v4.2+) | Yes (v4.2+) | No | No | No | No |
| ZPA reauthentication prompts | Yes | Yes | Not confirmed | Yes (in-app) | Yes (in-app) | Not confirmed |
| AUP overlay | Yes | Yes | Not confirmed | Yes | Yes | Not confirmed |
| captive portal notification | Yes | Yes | No | Yes | Yes | Not confirmed |
| `disableCaptivePortalNotification` install param | No | No | No | Yes | No | No |
| In-app status messages | Yes | Yes | Yes | Yes | Yes | Yes |

Notes:

- **Linux.** Linux is listed in the Web Policy per-platform sub-policies but the vendor notification doc does not address Linux for any of the notification framework features. Treat Linux as unsupported for ZCC-emitted OS toasts until confirmed.
- **iOS.** The install parameter `disableCaptivePortalNotification` (bool, default `0`) suppresses captive portal detection notifications at the MDM-config layer. Source: [`./install-parameters.md § UI / user access`](./install-parameters.md).
- **Android.** Notification delivery depends on Android OS version and whether the user has granted the ZCC app notification permission (Android 13+). MDM-deployed ZCC may have permissions auto-granted via managed configuration.
- **ChromeOS.** ChromeOS is listed in the Android/ChromeOS install parameter doc but the notification behavior is not separately described in the reviewed vendor notification doc.
- **Windows Toast.** On Windows 10/11, ZCC notifications appear in the Action Center if the user has not dismissed them. The Custom Timer setting (5–60 seconds) controls how long the toast banner is visible before collapsing to the Action Center.
- **macOS Notification Center.** Notifications from ZCC appear in Notification Center. If the user has set the ZCC app to "Banners" (not "Alerts") in System Settings, the notification auto-dismisses per the Custom Timer. If set to "Alerts," the notification persists regardless of Custom Timer, meaning the "Enable Persistent Notifications" behavior is partially OS-controlled.

## 5. Configuration via App Profiles

The End User Notifications settings described in Section 2 are global to the ZCC tenant (not per-App-Profile) when configured under **Administration > Client Connector Notifications**. However, some notification-adjacent behaviors are per-App-Profile:

- **Captive portal grace period.** The duration during which ZCC disables web security after detecting a captive portal is now configured per App Profile on newer tenants, not globally via the FailOpenPolicy object. See [`./forwarding-profile.md § Fail-open policy`](./forwarding-profile.md) and [`./web-policy.md`](./web-policy.md).
- **ZPA reauthentication period.** The global `Show ZPA Reauthentication Notifications Every (In Minutes)` setting may be overridden at the App Profile level. Exact per-profile field not confirmed from reviewed sources; see Deferred.
- **Platform sub-policies.** `windowsPolicy`, `macPolicy`, `linuxPolicy`, `iosPolicy`, `androidPolicy` sub-objects on the Web Policy (App Profile) include per-platform controls that can affect ZCC UI behavior (e.g., `disable_password`, `logout_password`), but do not include direct notification content fields. See [`./web-policy.md § Per-platform sub-policies`](./web-policy.md).
- **Notification Templates.** When notification templates are in use, they are configured at the ZCC Portal level (Administration > Client Connector Notifications > Notification Templates tab), not per-App-Profile. Whether templates can be scoped per App Profile is not confirmed from reviewed sources; see Deferred.

The ZCC SDK does not expose notification configuration as a discrete service. The `web_policy` service (`/zcc/papi/public/v1/web/policy`) manages App Profile data but does not include notification toggle fields in the Python or Go SDK models reviewed. Notification settings appear to be configured exclusively through the ZCC Portal admin UI. [Source: `references/zcc/sdk.md`]

## 6. Localization

The Zscaler Notification Framework supports English by default. Multi-language notification content through the Notification Templates system is referenced in related-article links in the vendor notification doc but was not captured in available sources. Specific details — supported languages, fallback behavior, per-locale template definition — are not confirmed from reviewed sources. See Deferred.

For the AUP message, the content is free-form HTML. Admins who need multi-language AUP text must embed all language variants in the HTML content field manually; there is no built-in language selector for AUP. [Source: `vendor/zscaler-help/configuring-acceptable-use-policy-zscaler-app.md`]

## 7. Disabling notifications

Admins may want a silent client — for kiosk deployments, server-side VDI sessions, lab environments, or any context where system-level pop-ups are disruptive or confusing to users.

**How to silence:**

- Set **Enable Notifications by Default** to off in the End User Notifications tab. Users enrolled after this change will not receive notifications by default. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`]
- Disable each category toggle (App Updates, Service Status, ZIA Notifications, ZPA Reauthentication) individually.
- On iOS: set the `disableCaptivePortalNotification` managed-config key to `1` to suppress captive portal notifications at enrollment time. [Source: `references/zcc/install-parameters.md`]
- On Windows VDI: set the `VDI` / `vdi` install parameter to `1`. ZCC adjusts behavior for virtual desktop scenarios. [Source: `references/zcc/install-parameters.md`]

**Risks of disabling:**

- Users will not see captive portal alerts and may be unable to complete portal authentication without admin assistance, especially in hotel/airport environments.
- ZPA reauthentication prompts will not appear; users whose ZPA session has expired will silently lose access to private apps without an explanation.
- Service Disaster Recovery notifications are suppressed; users in DR mode may not understand why access behavior has changed.
- If Enable Notifications by Default is off, users who need notifications must manually re-enable from within ZCC — which requires knowing to do so.

Silent-client configurations should document which notification categories are disabled so that support teams know to probe user-side ZCC status directly rather than assuming notifications were received.

## 8. Logging

ZCC notification events are not represented as distinct entries in the ZIA admin audit log or the ZPA audit log, based on reviewed sources. The following logging surfaces apply:

- **ZCC diagnostic logs.** The ZCC client writes local diagnostic logs that include notification delivery events. Log verbosity and rotation are controlled by `log_level`, `log_mode`, and `log_file_size` fields on the Web Policy object. [Source: `references/zcc/web-policy.md § Top-level ZCC behavior knobs`]
- **Disable-reason tracking.** When a user disables ZCC, the reason can be captured via `send_disable_service_reason` on the Web Policy, and aggregated via the `/downloadDisableReasons` API endpoint. [Source: `references/zcc/web-policy.md`; `references/zcc/sdk.md § devices`]
- **ZIA admin audit log.** Configuration changes to notification settings (e.g., toggling "Enable Notifications by Default") generate entries in the ZIA/ZCC Portal admin audit log. User-facing notification delivery itself is not represented in the audit log.
- **ZPA reauthentication events.** ZPA session expiry and reauth events are logged in ZPA analytics (accessible from the ZPA Admin Portal), not in the ZCC notification log. Cross-reference ZPA session logs if investigating why a user received or did not receive a ZPA reauth prompt.

Whether ZCC notification delivery events (specifically: "notification shown", "user dismissed", "user acknowledged AUP") are surfaced in any Zscaler cloud log feed is not confirmed from reviewed sources. See Deferred.

## 9. Common operator confusions

### Block-page text in ZIA is not notification text in ZCC

When a ZIA URL filtering rule blocks a URL, the user's browser receives an HTTP 403 response with an HTML block page rendered by the ZIA Public Service Edge. This block page is distinct from any ZCC notification. Operators who ask "how do I change the text the user sees when a site is blocked?" need to be directed to ZIA block page customization, not to the ZCC End User Notifications tab. See [`../zia/url-filtering.md`](../zia/url-filtering.md).

ZCC does not generate a pop-up notification for every URL block. URL block events are cloud-side; ZCC is the forwarding agent. The ZIA DLP notification is one exception — DLP events can be forwarded back through ZCC to the endpoint — but URL category blocks are not.

### AUP page is a ZCC overlay, not a ZIA block page

The AUP configured under ZCC Portal > Client Connector Notifications is displayed by the ZCC client itself before the user is allowed to connect. It is not a ZIA block page, not a ZPA policy enforcement page, and not a browser redirect. It is rendered in the ZCC application window. Admins who see AUP complaints and look in ZIA logs will not find relevant events there.

### ZIA DLP "notifications" and ZIA block pages are different

When "Enable ZIA Notifications" is on and ZIA DLP policy triggers, ZCC can surface a DLP notification as an OS-level toast. This is not the same as a ZIA block page for a DLP rule that has action=Block. A DLP notification through ZCC indicates the DLP event was logged or flagged but the traffic was not necessarily blocked — depending on rule action. If the DLP rule action is Block, the user sees a ZIA block page in their browser; the ZCC DLP notification is an additional alert layered on top, not a replacement.

### The Zscaler Notification Framework is Windows/macOS only at v4.2+

Operators who enable Persistent Notifications and Custom Timer settings expecting them to apply to iOS or Android endpoints will not see results on those platforms. The framework is Windows and macOS only as of ZCC 4.2. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`]

## 10. Operational gotchas

### OS notification permission denials

On Windows 10/11 with Focus Assist (now called "Do Not Disturb") or Notification Center muted, ZCC-emitted toast notifications may be suppressed by the OS regardless of ZCC configuration. The notification is delivered but silently placed in Action Center. On Android 13+, the OS requires explicit notification permission grant for each app; if the user denied permission during ZCC installation, no notifications will appear regardless of ZCC settings. MDM-managed deployments can pre-grant notification permissions via device configuration profiles, reducing this risk.

On macOS, the user must grant notification permission to ZCC in System Settings > Notifications. On managed Macs, MDM can push a Notifications preference plist to pre-approve ZCC. Without this, the first ZCC notification generates an OS permission prompt, which users in kiosk contexts may never see or accept.

### MDM constraints on iOS

On iOS, ZCC is delivered and configured via MDM managed-app-config. The `disableCaptivePortalNotification` install parameter is the only notification-related MDM key exposed in reviewed sources. Other notification behaviors on iOS are controlled by the ZCC Portal global settings, not by MDM. Operators expecting MDM to control all ZCC notification settings will find the surface area limited.

### Version skew with older clients ignoring new notification types

The Zscaler Notification Framework fields (Custom Timer, Persistent Notifications, ZIA Notification Persistent) require ZCC 4.2 or later on Windows and macOS. [Source: `vendor/zscaler-help/configuring-end-user-notifications-zscaler-client-connector.md`] Devices running older clients will not honor these settings. In a mixed-version fleet, the admin console settings apply only to clients that support the framework; older clients exhibit pre-4.2 default notification behavior. Fleet version reporting via `/downloadServiceStatus` (ZCC SDK `devices.download_service_status`) can identify out-of-date clients.

App profile updates — including notification settings — propagate to endpoints only on ZCC restart or user logout and re-login, not continuously. [Source: `references/zcc/forwarding-profile.md § Edge cases`] A setting change pushed in the admin console does not reach currently-connected devices until their next ZCC restart event.

### Captive portal notification and the grace period timer

When ZCC detects a captive portal, it triggers a persistent notification (if the framework is enabled and Persistent Notifications is on) and simultaneously starts the captive-portal grace-period timer (`captivePortalWebSecDisableMinutes`). If the user sees the notification but takes longer to complete the portal login than the grace period allows, ZCC re-enforces web security and the user is blocked — despite having received and acknowledged the notification. The notification and the timer run independently. Operators troubleshooting "user got the pop-up but still got blocked at the airport" should inspect the captive portal grace period setting on the user's active App Profile, not the notification configuration. See [`./forwarding-profile.md § Fail-open policy`](./forwarding-profile.md).

### ZPA reauthentication prompts and user-disabled toggle

Users can disable ZPA reauthentication notifications from within the ZCC client. This toggle is per-device and persists until the user re-enables it or ZCC is reinstalled. Admins have no API-visible record of which users have self-disabled this notification. If a user reports they are not seeing reauth prompts, check whether they previously toggled it off in ZCC before assuming a configuration issue.
