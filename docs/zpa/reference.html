At a glance

  • Default denyNo matching access-policy rule = blocked. Opposite of ZIA URL Filtering's default-allow.
  • Two-stage matchSegment selection runs client-side in ZCC. Policy evaluation runs in ZPA. Both must allow.
  • Specificity winsMost-specific FQDN selects the segment. Specificity ranks on domain only — port narrowness doesn't count.
  • Top-down rulesFirst-match-wins. Order matters. Once a rule fires, evaluation stops.
  • Outbound onlyApp Connectors dial out. Never accept inbound. No public endpoint to expose.
  • Group everythingApp Connector Groups bind to Server Groups bind to Segments. Upgrades, capacity, and policy all key off groups.
§ 0 · Evaluation order

How a ZPA request is decided

Four families of policy run in a fixed order against every connection. Operator misconfigurations almost always trace to mistaking which family is firing.

Fig. 1The order is fixed. A bypass at any earlier stage skips the ones to its right.

The two stages that decide whether traffic reaches an app at all are Forwarding (does ZPA handle this traffic?) and Access (is this user allowed?). The two narrower stages — Timeout and Inspection — refine an already-allowed session.

Common misconfiguration Mixing Forwarding BYPASS with Access DENY for the same app. The DENY never fires because forwarding bypassed ZPA first; users still reach the app directly through their default route.
§ 1 · Application segments

Application segments

A segment is a named bundle of FQDNs and ports plus the App Connector group that fronts them. The unit of application-scoping that everything else hangs off.

Cross-microtenant Move and Share

Two operations move or share segments across microtenant boundaries — both Go-SDK only.

  • AppSegmentMicrotenantMove — POST .../application/{id}/move
  • AppSegmentMicrotenantShare — PUT .../application/{id}/share

An operator asking “how do I move a segment to a different microtenant programmatically” needs the Go SDK or direct HTTP — Python has no equivalent.

Specificity-wins, and the carved-out wildcard

When two or more application segments cover the same destination address, ZCC picks the more specific one. Specificity ranks strictly on FQDN dimension:

server1.db.hr.acme.com  >  *.db.hr.acme.com  >  *.hr.acme.com  >  *.acme.com  >  *.com

IP equivalent: /32 host beats /24 subnet. Port narrowness does not enter specificity.

The consequence is the most common ZPA footgun:

The carved-out default Configuring a specific-FQDN segment removes that FQDN from any overlapping wildcard segment entirely. If the user's request matches a port the specific segment doesn't cover, ZCC does not fall back to the less-specific wildcard — it bypasses ZPA and sends the traffic direct.

Worked example — the port-mismatch footgun

Two segments cover www.example.com:

SegmentDomainPorts
A*.example.comTCP 1–65535, UDP 1–65535
Bwww.example.comTCP 8843

User requests www.example.com:80:

Most specific.Segment B wins (www.example.com beats *.example.com).
Port mismatch.Segment B's port list does not include 80.
Bypass — not fallback.ZCC drops the connection request to ZPA. Segment A is not consulted.
Resolution.Open TCP 80 on Segment B, OR enable Multimatch on the wildcard so it catches ports the specific segment doesn't cover.

Where the traffic goesThe help docs phrase this two ways: “ZCC bypasses ZPA and sends traffic directly” and “the connection is dropped by ZPA.” Same event — ZPA is out of the path. Whether the request succeeds against the destination, hits a perimeter firewall, or fails to resolve depends on the client's forwarding profile and network egress, none of which is ZPA's concern.

Multimatch (INCLUSIVE mode)

Multimatch lifts the carve-out: a segment with match_style = INCLUSIVE allows a request to match multiple segments simultaneously. The wildcard segment then catches ports the specific segment doesn't cover.

Validated at config time, not traffic time

ZPA evaluates Multimatch across all segments containing overlapping domains when an admin enables or disables it. If a domain appears in multiple segments with different Multimatch settings, the change is rejected at save. You cannot get into a mixed-style state.

The specificity stack

Multimatch is itself ordered by specificity. As soon as a segment in the stack has Multimatch disabled, the descent stops. Example for server1.db.hr.acme.com:

SegmentMultimatchMatches?
*.db.hr.acme.comEnabledYes
*.ui.hr.acme.comDisabledNo (irrelevant — different domain)
*.hr.acme.comEnabledYes
*.acme.comDisabledNo — descent stops here

Disallowed combinations

Multimatch must be disabled if the segment uses any of:

  • Browser Access
  • AppProtection
  • Privileged Remote Access
  • Double Encryption
  • ZIA inline inspection (inspect_traffic_with_zia)
  • Source IP Anchor (SIPA)

And if Multimatch is enabled, health_reporting can only be ON_ACCESS or NONE — not CONTINUOUS.

Connector and client requirements

  • App Connectors / Private Service Edges on 24.298.1+
  • ZCC: Windows 4.6.0.282+ or 4.7.0.88+; macOS 4.5.2.98+; iOS 4.4.1+ (with “Use Tunnel SDK Version 4.3 or above”); Android 3.10+; Linux 4.2+

Bypass precedence

The Bypass setting on a segment outranks both client forwarding policy and the specificity tie-break:

Bypass = Always
If any segment covering the destination has bypass_type = ALWAYS, that segment wins. Traffic forwards direct, does not enter ZPA. Overrides specificity-wins and forwarding-policy rules.
Bypass = On Net
Bypass only when the user is on the corporate network. Undocumented in most help articles; valid at the API.
Bypass = Never
Default. Specificity-wins and forwarding policy decide.

Edge cases & quirks

ICMP asymmetric aggregation
If ICMP is enabled on one segment for an app and disabled on another segment for the same app, ZCC considers ICMP enabled and forwards. Cross-segment ICMP is OR-aggregated, not most-specific-wins.
IdP overlap
An IdP defined as an application must have Authentication Timeout = Never. If the IdP domain overlaps with an app-discovery domain, you must bypass the IdP domain in ZPA to avoid re-auth failure loops.
CNAME on macOS / iOS
is_cname_enabled controls CNAME DNS records from App Connectors to ZCC. Not applicable to Browser Access. Best practice: disable on macOS / iOS when the app is accessed by CNAME.
Browser Access constraints
Port range must include the BA-specified port. TCP ports only.
Double Encryption mutex
Cannot combine with Browser Access or Source IP Anchoring on the same segment.
SIPA cross-product gate
Source IP Anchor on a segment opts into the SIPA flow — ZIA-inspected traffic forwards to this segment's App Connector so the destination sees a customer-controlled IP. Mutex with Browser Access, Double Encryption, and Multimatch on the same segment. Also: only SIPA-enabled segments appear in ZIA's zpa_app_segments SSL-inspection-rule selector — non-SIPA segments are invisible to it.
Python SDK clientless_app_ids hazard
The SDK checks if "clientless_app_ids" in body:, not if body.get("clientless_app_ids"):. Including clientless_app_ids: None in the body triggers a Browser-Access lookup, which fails with “No matching clientless App found” for a standard segment. Omit the key entirely; passing None is not equivalent. Affects both create and update paths.
Same-FQDN, same-Bypass tie-break
Two segments with identical domain_names and same Multimatch / Bypass settings: behavior is not documented. Save-time validation likely blocks the configuration. If both saved anyway (older rules, or save-time validator bypassed), ZCC picks one deterministically per session — but which one is unspecified. clarification zpa-04
§ 2 · Access policy

Access policy precedence

Top-down, first-match-wins. The first rule whose criteria all evaluate true fires. No matching rule = blocked.

From About Access Policy “Private Access evaluates access policy rules using the most specific application segment and a top-down, first-match principle.”

This decomposes into two ordered stages: segment selection (covered above, runs in ZCC) picks the segment; policy evaluation (runs in ZPA) walks the rule list top-down against rules referencing that segment, directly or via a Segment Group.

Criteria

Available criteria on a rule:

CriterionWhat it evaluates
Application SegmentsSpecific segments the rule applies to.
Segment GroupsGroups of segments the rule applies to.
Client TypesZCC, ZCC for VDI, ZCC Partner, Branch Connector, Cloud Connector, Machine Tunnel, Web Browser, Internet & SaaS Service Edge, Extranet.
Posture ProfilesDevice posture checks reported by ZCC. POSTURE condition's rhs is the posture_udid, not the profile name or ID.
Trusted NetworksTrusted-network state from ZCC.
Locations · Branch Connector · Cloud Connector Groups · Machine GroupsNetwork-grouping criteria.
Country CodesCountry of the user's IP. Uses the last NATed layer-3 public IP, not the user's original IP — relevant for SIPA / multi-hop / Cloud Connector cases.
PlatformsWindows, macOS, Linux, Android. RHS is string "true"/"false", not boolean — wire-format quirk.
SAML / SCIM / SCIM GroupIdP attributes. Use entry_values with {lhs, rhs} pairs, not values.
Risk ScoresExternal risk source (e.g., Internet & SaaS). The cross-product hook bringing ZIA browsing-risk data into ZPA decisions.
Chrome Enterprise Browser · Workload Groups · ExtranetLess common scoping criteria.

Boolean operators

Within one criterion
Multiple values are ORed. Sales OR QA Site OR Internal Services.
App Segments vs Segment Groups
ORed. A user matching either branch satisfies the app-scope.
Across distinct criteria
ANDed. SAML ∧ Country ∧ Client Type all required.
Operands within one block
OR. Mixed types allowed: SAML user OR SCIM_GROUP membership in one block.
Multiple condition blocks
AND. Block 1 ∧ Block 2 ∧ Block 3.

So a rule structurally looks like:

(app_segments OR segment_groups)  AND
(saml_attrs)                       AND
(countries)                        AND
(client_types)                     AND
...

Rule actions

The authoritative list lives in Configuring Access Policies; About Access Policy uses informal aliases. Prefer the former:

ActionMeaningNotes
Allow AccessAllow access to applications.Default action.
Block AccessBlock access.Optional Pop-Up Message to User. Visible only on Block; ZCC v4.7+ Windows only.
Conditional AccessStep-up authentication. Forces additional auth match before granting access.Requires ZIdentity. OIDC IdP only — SAML doesn't support step-up.

Terminology driftAbout Access Policy calls one of these “Require Approval.” That's an informal name for either Conditional Access or Allow with Privileged Approval — not a separate action. clarification zpa-06

Order & first-match

Rules evaluate top-down. The first one whose criteria all match fires; the rest are skipped. This is the source of nearly all “why isn't my block rule blocking” tickets.

Conflicting-rule examples

OrderRuleEffect on Sales user
Rule 1Allow any user → any appAllowed — fires first
Rule 2Block Sales → any appSkipped

Sales is not blocked. Fix: reorder so the Block rule comes before the broad Allow.

OrderRuleEffect on Marketing user
Rule 1Allow Marketing → all appsAllowed — fires first
Rule 2Deny Marketing → Operations AppsSkipped

Marketing still has access. Same fix.

Under Multimatch

Multiple segments match the same request. Their rules are interleaved into the same top-down evaluation. A block on the more-specific segment can fire before an allow on the wildcard, even when the wildcard rule appears later. Worked example from Using Application Segment Multimatch:

Segments: Wildcard_AS (*.example.com, TCP 1–52 + 54–65535, MM on) and Server1_AS (server1.example.com, TCP 443 + 3389, MM on).

OrderRule
Rule 1Allow Server1_AS for Admin_Grp
Rule 2Block Server1_AS for All
Rule 3Allow Wildcard_AS for All
UserRequestMatched rule
user1 (Admin)server1.example.com:3389Rule 1 (Allow)
user1server1.example.com:443Rule 1 (Allow)
user1server1.example.com:22Rule 3 (Allow Wildcard)
user3 (IT)server1.example.com:443Rule 2 (Block)

Programmatic reordering

Two Go-SDK endpoints; Python lacks both:

Reorder
PUT .../rule/{id}/reorder/{newOrder} — move a single rule.
BulkReorder
Accepts a full ordered rule-ID list and applies the entire reordering atomically. Useful after imports or large refactors.

Hidden rules at the top

Deception rules occupy rule_order = 1 If Zscaler Deception is enabled in the tenant, an immutable Deception rule sits at order 1 — invisible to the admin console as editable, but the API includes it in order calculations. TF-managed rules must start at rule_order = 2 and stay sequential. Gaps in rule_order cause Terraform to reorder unpredictably (issue #166, fixed v2.3.1).

The four policy families

Forwarding Policy
Decides whether ZPA handles the traffic at all. BYPASS sends direct, skipping access-policy entirely. INTERCEPT keeps it on ZPA. INTERCEPT_ACCESSIBLE is a hybrid — route through ZPA only if reachable via App Connectors; otherwise direct. Empty conditions list = global rule.
Access Policy
Allow / Deny / Conditional Access / Allow-with-Privileged-Approval. Only evaluates if Forwarding kept traffic on ZPA. Empty conditions list = allow all to all apps.
Timeout Policy
Independent of Allow/Deny — governs session re-auth timing. Action: RE_AUTH. Defaults: 2 days session, 10 minutes idle. Values are strings: "N Minutes", "N Hours", "N Days", "Never". Restricted condition subset — COUNTRY_CODE, TRUSTED_NETWORK, CLIENT_TYPE are not supported.
Inspection Policy (v2)
INSPECT or BYPASS_INSPECT. Controls whether AppProtection inline inspection runs. Has its own narrower condition set, distinct from access/forwarding.

Re-auth fields and ForceNew

Two distinct timeout fields on access rules. Both are ForceNew: changing either on an existing rule requires destroy-recreate at the API. The rule is deleted and re-created, which can shuffle order and cause transient policy gaps. Plan changes for maintenance windows.

reauth_timeout
Session duration before full re-auth. Default None → tenant global applies.
reauth_idle_timeout
Idle duration before re-auth. Default None → tenant global applies.

Action-specific profiles, conditionally relevant

Two profile fields are only meaningful with specific actions; the SDK does not validate the cross-field dependency:

zpn_isolation_profile_id
Only relevant when action = ISOLATE. References a ZPA isolation profile (Turbo Mode, copy/paste allow, region, etc.).
zpn_inspection_profile_id
zpn_inspection_profile_name
Only relevant when action = INSPECT.

Setting an irrelevant profile passes silently; the API's behavior in that case is unstated. Debugging “why isn't my isolation profile applying” → verify the action actually is ISOLATE.

Edge cases & quirks

SCIM sync timing
After enabling SCIM, ZPA checks the SCIM database to allow/block. Enabling SCIM policies before sync completes denies unrecognised users by default. Wait 48 hours minimum after enabling SCIM (sometimes up to a week) before activating SCIM-criterion policies.
Country criterion under SIPA
Country is the last NATed public IP's country. For SIPA traffic, that's the ZIA Public Service Edge's country, not the user's. Country-scoped ZPA rules misfire for SIPA users.
Deception policy ordering
Regular rules must have order > Deception rules. Copy/edit/delete are disabled on Deception-configured rules — they are managed from the Deception Admin Portal, not the ZPA console. Editing out-of-band desynchronises Deception's view; an attacker's activity then shows up as a normal admin audit event instead of triggering a Deception alert. clarification zpa-07 (resolved)
Default block
No matching rule = blocked. Until you configure rules for applications, users are blocked from accessing any applications or segment groups.
devicePostureFailureNotificationEnabled
Controls user notifications on posture-based denials. Wire/snapshot key is camelCase; Python SDK exposes it snake-cased. Operators querying snapshot JSON must use camelCase.
rule_order vs priority
Different fields. rule_order controls evaluation position. priority is metadata, sometimes used by the admin console for grouping. Setting them inconsistently doesn't error but confuses readers — keep them aligned (rule_order = N, priority = N).
§ 3 · App Connector

App Connector

The VM at the application end of every ZPA flow. Outbound-only; receives requests from a ZPA Service Edge, forwards them to the actual app, streams responses back.

Architecture & placement

Outbound only
Never accepts inbound from the internet. The structural property that makes ZPA safe to deploy without exposing internal apps.
Typical placement
DMZ, or any segment with connectivity to both the apps and the ZPA cloud. Data centre, cloud VPC, branch with on-prem apps.
Always active in groups
All connectors in a group serve traffic simultaneously. No active/passive. ZPA selects the closest one per request based on user location and connector-to-app latency.
Independent enrolment
Connectors never communicate with each other. Each enrols separately via provisioning key + TLS client cert.
Sizing
N+1 redundancy. Adding connectors to a group increases capacity. Sizing depends on concurrent users, app throughput, and inspection feature set (Double Encryption, AppProtection add overhead).

The grouping chain

App Segment  →  Server Group  →  App Connector Group  →  App Connectors (N)
                                                          (each with own cert)

Same App Connector Group can serve many segments. Provisioning key determines group assignment at enrolment.

Silent partial-config failure An App Connector Group with no Server Group association routes no traffic — and the admin console doesn't flag the partial config as invalid. Same applies to network reachability: only associate groups with apps the connectors can actually reach.

Provisioning keys

A long pipe-delimited string that functions as the shared-secret credential for connector enrolment.

1|api.private.zscaler.com|68F0AOEgpcG8McLmwdborq2m6v2A5oNEpSztJ/...(payload)...
Bound to one group
Each key enrols connectors into a single App Connector Group. Cross-group moves require re-enrolment with a different key.
Reuse-count tracked
Set Maximum Reuse of Provisioning Key at creation. ZPA tracks utilisation; when count hits max, key stops working for new enrolments.
Multiple keys per group
Useful for separating deployment pipelines (prod vs DR) so utilisation logs differentiate.
Treat as a secret
If View or Export Provisioning Key After Creation was disabled at creation, the key is never retrievable. Backup must exist externally (vault) or generate a new key.
The #1 enrolment failure Provisioning-key utilisation count hitting its max. Symptoms: new connectors fail to enrol; existing ones keep working. Diagnostic: check the key's utilisation count against its configured max.

Enrolment failure log signature

An exact paste of this in a support ticket narrows diagnosis to “key wrong, truncated, or utilisation-exhausted”:

notice:Checking Enrollment
notice:No valid certificate. Attempting to enroll
notice:Enroll: Connecting to api.private.zscaler.com via co2br.prod.zpath.net.
error:Login request failed - http status(401) nonce(<3|api.private.zscaler.com|0/Z6lDT...>) fingerprint(<oXaN4RRiMc...>)
notice:Certificate enrollment failed.

TF data-source asymmetry

The TF resource accepts five association_type values; the matching data source accepts only two. Operators creating keys for the “extended” types (EXPORTER_GRP, NP_ASSISTANT_GRP, SITE_CONTROLLER_GRP) must look up by ID, not by association-type-plus-name through the data source. Auto-discovery tooling will silently miss these keys.

Software updates

Group-level scheduling
Admin specifies day + time at the App Connector Group level.
4-hour rolling window
ZPA picks one connector at a time, upgrades it (restart + reconnect), then the next. The group stays available because only one is down at a time.
Window is not extendable
Groups too large to finish in 4 hours stagger across multiple nights. Pick upgrade nights carefully.
Failed updates retry next window
Until every connector is on target version.
Self-upgrade on first boot
Starting with App Connector 24.650.4+, a version check runs on initial connection. Freshly-deployed connectors may self-upgrade before the scheduled window.
StatusMeaning
ScheduledIn queue for next periodic update window. Hover shows scheduled version + 4-hour window.
SuccessOn target version.
FailureUpdate attempted and failed. Restart the VM. Retries next window.
OS updates are the customer's job Zscaler doesn't patch the underlying kernel/distribution. App Connector software is designed to be host-OS-update compatible, but a long-running VM with un-patched OS is a common audit finding.

Edge cases & quirks

VM-cloning fingerprint collision
Cloning a VM template without unique re-enrolment leaves all clones sharing a hardware fingerprint. ZPA detects the collision and disables all but one. Remedy: re-enrol each clone with a unique fingerprint.
Latency-based selection isn't pure geo
ZPA uses continuous app-to-connector RTT measurements, not static geography. A connector that's geographically far but has a fast link to the app can be preferred. Counter-intuitive for operators expecting purely geographic routing.
Trailing whitespace in copied keys
Keys copied from the portal UI can silently include trailing whitespace. Pasting into a deployment template fails enrolment with the 401 above. Trim whitespace.
Private Service Edge service_edges drift
The service_edges block on zpa_service_edge_group is documented as optional but is required if any Service Edges actually attached to the group. Omitting it makes TF try to detach them on every apply. PSE UID isn't visible in the portal at registration — capture it via API or terraform import, then add to the block. Provider versions v4.0.9–v4.1.2 had drift bugs; v4.1.3 stabilised.
LSS retransmit window is shorter than NSS
Connectivity gap between Private Access and the App Connector → LSS retransmits at most 15 minutes, and delivery is not guaranteed. Connectivity gap between App Connector and SIEM → no retransmit at all (audit logs excepted). Operators familiar with NSS's 60-minute opt-in recovery often assume LSS matches; it doesn't. A 30-minute App Connector outage = roughly 15 minutes permanent log gap.

Common question shapes

Where to start when an operator describes a symptom.

SymptomLikely causeStart here
“Why is this user reaching app X directly, not through ZPA?” Specificity-wins carve-out: a more-specific segment covers X but doesn't cover the requested port. ZCC drops to direct. §1 Specificity-wins
“Block rule isn't blocking.” An earlier Allow rule fires first. §2 Order & first-match
“Block rule isn't blocking, but no Allow rule above it covers this user.” Forwarding Policy BYPASS ran first; access policy never evaluated. §0 Evaluation order

“Conditional Access action does nothing.” SAML IdP. Step-up requires OIDC. §2 Rule actions
“Multimatch save is rejected.” Overlapping domain in another segment with the opposite Multimatch setting. ZPA validates at config time. §1 Multimatch
