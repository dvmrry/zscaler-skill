ZIA is a cloud-delivered inline proxy with default-allow posture. Traffic passes through a staged pipeline — Firewall, then a two-pass Web module bracketing an SSL Inspection decision. This reference covers the three modules where misconfiguration most concentrates: URL Filtering, SSL Inspection, and the evaluation order that connects them.

Default posture

ZIA's terminal behavior at every policy stage is allow. Traffic that matches no URL Filtering rule, no CAC rule, and no Firewall rule passes. This is the exact inversion of ZPA's default-deny and is the root cause of a recurring class of over-permissive configuration.

URL Filtering

Rule anatomy

URL Filtering rules evaluate ascending by order (first-match-wins). The rule list terminates with a predefined default rule that allows all unmatched traffic. Every rule carries a rank (1–7) that is stamped at creation and controls which admins can edit it — this is distinct from order.

Category resolution and wildcard semantics

URL-to-category assignment happens before any rule evaluates, by a specificity-wins lookup across all custom categories. Rule order does not affect which category a URL lands in — only which rule fires after the category is resolved.

Wildcard forms

Leading-period form (.example.com) is the only valid wildcard. It matches the bare domain, any subdomain up to 5 levels deep, and any path under any of those (implicit right wildcard).

*.example.com and *example.com are explicitly documented as invalid. The admin console may accept asterisk entries without error — verified operator report (2026-04-23) — but what is actually stored or matched is unverified. Treat asterisk entries as broken; translate to leading-period form before saving.

Specificity order for category resolution

When a URL matches entries in multiple custom categories, the most specific entry across all categories wins — regardless of which rule references which category:

  1. Exact: atlanta.example.com
  2. Leading-period subdomain: .atlanta.example.com
  3. Leading-period parent: .example.com
A URL assigned to Category B by a more-specific entry will trigger the Category B rule — even if a Category A rule (referencing a broader entry) is higher in the rule list. Category assignment is not a property of rule order.

Additional category gotchas

Keywords match the entire URI. A keyword like gambling in a custom category matches www.google.com/search?q=gambling — the entire URI, not just the hostname. Test keyword categories against search engine queries before deploying.

TLD-only blocking requires a separate category type. Regular custom categories reject .gov, .io, etc. as invalid. Use type = "TLD_CATEGORY" with entries in dbCategorizedUrls, not urls. This type is not surfaced in the admin console.

Regex patterns are API-only. regexPatterns and regexPatternsRetainingParentCategory are writable on custom categories via API/SDK — not visible in the console. A category with entries that don't match documented formats likely contains regex patterns.

Bulk URL add is all-or-nothing. If any single entry in a batch update is invalid format, the entire batch is rejected. Validate all entries pre-submission in automation.

Admin Rank and Cascading

Admin Rank

Admin Rank (1–7) is assigned to each admin and stamped on each rule at creation. Two effects:

  1. Edit gate: an admin can only create or modify rules whose rank ≤ their own rank. A rank-5 admin cannot touch a rank-7 rule.
  2. Order constraint: higher-rank rules are constrained to appear before lower-rank rules in the evaluation order. Rank-7 rules must precede rank-5 rules.

Admin Rank is a tenant-wide toggle: enableAdminRankAccess in advanced-settings.json. When off, rank constraints on ordering are relaxed and the edit gate does not apply.

The rank field in snapshot JSON is the rule's stamped rank; order is its evaluation position. These are separate integers — rank does not determine order directly, only constrains the valid range of order values the rule can occupy.

Cascading to URL Filtering

Cloud App Control (CAC) evaluates before URL Filtering. A CAC rule that allows a transaction short-circuits URL Filtering by default — URL Filtering never runs. The Cascading flag reverses this for Allow paths:

SettingEffect
Tenant-wide cascadeUrlFiltering = false (default)CAC Allow → URL Filtering does not run
Tenant-wide cascadeUrlFiltering = trueCAC Allow → URL Filtering evaluates and can block
Per-rule cascadingEnabled = true on a CAC ruleURL Filtering evaluates after that specific rule's Allow action, regardless of tenant-wide setting

CAC Block is irreversible regardless of cascading — a CAC block ends the transaction unconditionally. "No matching CAC rule" (CAC default Allow-All) routes to URL Filtering normally — the CAC default is not an explicit CAC Allow and does not trigger the short-circuit.

SSL Inspection

The two-pass model

SSL Inspection does not wrap URL Filtering — it sits between URL Filtering's two evaluation passes. The common question "does SSL happen before or after URL Filtering?" requires a three-stage answer.

  1. SNI pass (stage 02): URL Filtering evaluates on the destination domain — from the CONNECT header (explicit proxy) or SNI (transparent). A Block rule here ends the connection before the TLS handshake. ATP, CAC, URL Filtering, and Bandwidth run.
  2. SSL decision (stage 03): SSL Inspection policy evaluates. Outcome: Decrypt, Do Not Inspect (Evaluate), Do Not Inspect (Bypass), or Block.
  3. Full-URL pass (stage 04, Decrypt path only): URL Filtering evaluates again on the complete URL, method, headers, and body. DLP, Sandbox, Malware, File Type, and content-based IPS run. CAC can identify specific apps (e.g., docs.google.com vs drive.google.com).

Zscaler acts as a short-lived intermediate CA at stage 03. On decrypt, the Public Service Edge issues an ephemeral per-connection certificate signed by the configured intermediate CA. Two options: the default Zscaler-managed CA, or a customer's own PKI intermediate (requires additional subscription). Traffic content is processed in-memory — never written to disk.

QUIC (HTTP/3) bypasses TLS inspection — it uses UDP and skips the TCP handshake that interception depends on. Recommended mitigation: block QUIC at the firewall, forcing browsers to fall back to TCP/TLS. Without this, users on HTTP/3-capable browsers will have all HTTPS to QUIC-supporting destinations pass uninspected.

Decrypt vs bypass paths

The most consequential decision in SSL Inspection is not which traffic to decrypt but which variant of "Do Not Inspect" to use. The two variants have radically different downstream effects.

The predefined Zscaler Recommended Exemptions rule uses "Evaluate Other Policies." The predefined Microsoft 365 Click-to-Run rule uses "Bypass Other Policies." Operators who copy the wrong predefined rule as a template replicate the wrong variant silently. Always check action.doNotDecryptSubActions.bypassOtherPolicies when cloning a Do Not Inspect rule.

What Do Not Inspect (Evaluate) still covers

At stage 02 (SNI pass), URL Filtering and ATP can still block based on the destination domain before the SSL decision fires. A URL Filtering Block on a known-bad hostname works even with "Evaluate Other Policies" — the block fires at stage 02 before stage 03 makes the SSL decision. The SNI-pass block is TCP-level (before decryption) — it closes the CONNECT tunnel or sends a TCP reset.

Transparent vs explicit forwarding changes what SSL rules can match

In explicit proxy mode (PAC/ZCC), SSL rules evaluate on SNI only. In transparent mode (GRE/IPSec), SSL rules evaluate on SNI and destination IP. An SSL Do Not Inspect rule scoped to "Miscellaneous or Unknown" under transparent forwarding silently exempts nearly all traffic — most public IPs are categorized as Miscellaneous. The same rule under explicit forwarding only affects the small subset of HTTPS destinations with no SNI.

Policy Evaluation Order

Full pipeline

The complete ZIA evaluation order for outbound HTTPS, from connection attempt to allow/block. A block at any step halts that pass — subsequent steps do not run.

Stage 01 — Firewall module

All web traffic passes through the Firewall module first. IP/port rules, DNS Control, and L4 IPS. A Firewall Allow does not grant a web-module pass — both layers must independently not-block. A Firewall Block is unconditional regardless of what web policy would do.

DNS Control is inside the Firewall umbrella. Traffic from locations not onboarded to firewall forwarding never reaches the DNS Control engine — the rule list looks correct but the engine is bypassed silently.

Stage 02 — Web module, SNI pass

Evaluates on the destination domain from the CONNECT header (explicit) or SNI (transparent). Four modules run in this order:

  1. Known Malicious URLs (ATP)
  2. Cloud App Control
  3. URL Filtering
  4. Bandwidth Control

DLP, Sandbox, Malware, File Type, Browser Control, Country Blocking, and content-based IPS are deferred to stage 04.

Stage 03 — SSL Inspection decision

SSL policy evaluates. Outcome: Decrypt → stage 04 runs. Do Not Inspect (Evaluate) → stage 04 runs at SNI fidelity. Do Not Inspect (Bypass) → pipeline exits here. Block → connection terminated.

Stage 04 — Web module, full-URL pass (Decrypt path only)

Three sub-phases:

Sub-phaseOrder within phase
GET phaseCustom Malicious URLs → Cloud App Control → URL Filtering → Security Exceptions → Browser Control → Country Blocking → IPS (L7) → Suspicious Content → P2P → Bandwidth
POST phaseGET phase order + Malware → File Type → DLP
Response phaseSandbox → AI/ML Content Categorization

IPS (L7 / signature-based) is step 6 of 9 in the GET phase — this is why content-based IPS requires decryption. URL Filtering is step 3.

DNS Control fires on the DNS query before the HTTP request exists. It is not part of stages 01–04 in the web pipeline — it's a parallel module in the Firewall umbrella that runs at DNS resolution time. A URL Filtering block on badsite.com does not prevent DNS resolution; a DNS Control block prevents resolution and therefore access. Full enforcement requires both.

Module interactions

ModulesInteraction
URL Filtering × SSL InspectionStage 02 URL Filtering (domain-level) can block before SSL decides. "Bypass Other Policies" removes stage 04 URL Filtering entirely. "Evaluate Other Policies" allows stage 04 at SNI fidelity only.
