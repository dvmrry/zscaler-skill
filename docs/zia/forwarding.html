ZIA Traffic Forwarding

How traffic reaches a ZIA node from branch offices and on-premises networks — the infrastructure layer that must be correct before any ZIA policy runs.

At a glance

  • The problemZIA policy runs inside Zscaler's cloud. Traffic must physically arrive at a ZIA Public Service Edge (PSE) before any rule can apply. The forwarding method determines how it gets there.
  • GRE tunnelRouter encapsulates internet-bound traffic in GRE and sends it to a ZIA VIP. No encryption. Requires a static public egress IP. Primary branch on-ramp for high-throughput, fixed-location sites.
  • IPSec tunnelEncrypted tunnel from router/firewall to ZIA PSE. Use when the path traverses the public internet or when a static IP is unavailable (UFQDN credential type supports dynamic IPs).
  • PAC fileBrowser-level proxy selection via a JavaScript function. Directs browser traffic to ZIA's explicit proxy. No agent required; web traffic only; BYOD and guest-network use cases.
  • ZCCPer-endpoint agent. Z-Tunnel 1.0 (explicit proxy) or Z-Tunnel 2.0 (all-protocol DTLS/TLS tunnel). For mobile, remote, and off-network users.
  • Location = anchorEvery forwarding method surfaces inside ZIA as a Location. The Location is the primitive that all ZIA policy rules scope by. Get this right first.
  • Forwarding controlPost-inspection egress routing. Decides where ZIA sends traffic after inspecting it — to the internet, through ZPA (SIPA), to a proxy chain, or drops it. Distinct from how traffic arrives.

§ 1 · Locations

Locations

A Location is ZIA's forwarding-grouping primitive — it binds a forwarding path (GRE tunnel, IPSec SA, or egress IP) to a named policy scope. Every ZIA policy rule that matters to branch traffic scopes by Location or Location Group.

When ZIA receives a packet via GRE or IPSec, it identifies the Location by matching the tunnel's source IP against registered static IPs. For PAC-forwarded browser traffic, identity is established via user authentication or Surrogate IP. The Location is resolved before any content-inspection rule runs.

The three-tier model

Location Group       ←  what policy rules scope by
   ⌈— Location       ←  forwarding method + egress binding
   ⌈    ⌈— Sublocation (corporate)
   |    ⌈— Sublocation (guest)
   |    ⌈— Sublocation (other)    ← auto-created catch-all
   ⌈— Location
        ⌈— ...

Each Location carries per-location toggles that affect the entire policy stack for traffic arriving through it: Enable SSL Inspection, Enforce Authentication, Enforce Firewall Control, Enforce Bandwidth Control, Enable AUP, Use XFF from Client Request. Toggling any of these can silently change which dynamic Location Groups the Location belongs to, and thus which rules apply to its traffic.

Sublocations

A sublocation is a subdivision inside a Location, identified by an inner IP range (the encapsulated source IP inside the GRE/IPSec tunnel, or the XFF header). Sublocations allow guest and corporate traffic arriving through the same tunnel to receive different URL Filtering, authentication, and bandwidth policies.

Two sublocations are auto-created for every Location:

NamePurpose
otherCatch-all for inner IPs not matched by any explicit sublocation range. Receives default sublocation policy — often overlooked.
other6IPv6 equivalent. Created when IPv6 is enabled on the parent Location.
The “other” sublocation is not a no-op Any IP arriving from the Location that doesn’t match an explicit sublocation range falls into other. If you tighten policy on named sublocations but leave other at defaults, traffic from unmatched IPs escapes the tighter rules silently.

Sublocation IP ranges cannot overlap within a Location. The same IP address can exist in multiple Locations (different egress paths). Bandwidth at the sublocation level cascades to the parent — unused sublocation bandwidth is shared back to the parent pool, so sublocation limits are not hard-isolation boundaries.

Location Groups

Location Groups are policy-scoping containers. Rules reference groups rather than individual Locations. A Location or sublocation can be in multiple groups simultaneously — manual and dynamic groups can overlap.

TypeMembershipCap
ManualExplicitly assigned. Up to 32,000 locations/sublocations per group.32K members
DynamicAttribute-based; auto-updates. Conditions are AND-ed across attributes, OR-ed within an attribute’s value list.No member cap

Zscaler ships five predefined dynamic groups keyed to the Location Type field: Corporate User Traffic Group, Guest Wifi Group, IoT Traffic Group, Server Traffic Group, and Workload Traffic Group. Setting Location Type on a newly-created Location silently enrolls it in the matching predefined group at write time — the Location immediately inherits all rules scoped to that group.

Dynamic group rule design Scope policy rules by dynamic Location Groups rather than individual Locations. A new branch Location with the right attributes then inherits policy automatically without editing any rule.

XFF and sublocation matching

The Use XFF from Client Request toggle lets ZIA attribute traffic to a sublocation using an X-Forwarded-For header rather than the tunnel’s inner IP. This is useful when an upstream proxy or CDN terminates connections and inserts XFF carrying the original client IP. XFF is opt-in per Location because accepting unauthenticated headers is a spoofing risk.

§ 2 · GRE tunnels

GRE tunnels

The primary branch on-ramp for high-throughput fixed-location sites. GRE encapsulates internet-bound packets with a 4-byte header and sends them to a Zscaler VIP — no encryption, minimal overhead, 1 Gbps per tunnel when inner IPs are not NATed.

When to use GRE

GRE is appropriate when: the branch router or SD-WAN supports GRE; the path to Zscaler traverses a private network (MPLS, SD-WAN overlay) where in-flight encryption is not required; throughput is at or above 500 Mbps; and the site has a static public egress IP.

For sites requiring encrypted tunnels (public-internet path) or dynamic IPs, use IPSec instead.

Static IP requirement

Zscaler matches inbound GRE traffic to a Location by the tunnel’s source IP. This IP must be static — a dynamic IP causes the Location binding to fail when the address changes. The static IP is registered as a TrafficStaticIP object and then referenced by the GRE tunnel and its parent Location.

SDK lookup for available IPs: client.zia.traffic_static_ip.list_static_ips(query_params={"available_for_gre_tunnel": True}) — filters to IPs not yet assigned to a tunnel.

Bandwidth limits

ConditionMaximum per tunnel
Inner endpoint IPs not NATed1 Gbps
Inner endpoint IPs NATed250 Mbps

The NATed ceiling exists because Zscaler load-balances GRE traffic by encapsulated inner source IP. When inner IPs are NATed, all traffic from a site appears to come from a single source address, preventing the load balancer from spreading load across nodes. Sites needing more than 1 Gbps require multiple tunnels from distinct public source IPs (2 Gbps = 2 primary + 2 secondary tunnels). When running multiple source IPs for throughput scaling, maintain client-session persistence at the ECMP or load-balancer layer to prevent mid-session source-IP changes.

MTU and MSS clamping

GRE adds a 4-byte header over IP (20 bytes). For a 1,500-byte WAN interface:

WAN MTU  = 1500
GRE MTU  = 1500 - IP(20) - GRE(4) = 1476
GRE MSS  = 1476 - IP(20) - TCP(20) = 1436

Set MSS clamping on the GRE tunnel interface to 1436. Misconfigured MSS is the canonical cause of “browsing works but large file transfers hang” under GRE.

MSS vs MTU MTU is the maximum packet size on the link. MSS is negotiated in TCP and determines the largest segment the sender will generate. Clamping MSS at the tunnel interface prevents oversized segments that fragment after encapsulation.

VIP selection and datacenter routing

Each GRE tunnel references a primary_dest_vip and secondary_dest_vip — Zscaler VIP objects identifying the destination datacenter. The SDK auto-selects the closest geographically diverse VIP pair if none is specified at tunnel creation (get_closest_diverse_vip_ids). The within_country flag restricts VIP selection to datacenters in the same country as the tunnel source IP. Premium or surcharge datacenters are excluded from the VIP picker UI and require a Support ticket to enable.

The ip_unnumbered flag supports SD-WAN automated provisioning where an internal IP address is not assigned to the tunnel interface.

High availability and health monitoring

The recommended HA model uses two GRE tunnels per Location — primary to datacenter A, secondary to datacenter B in a different city. GRE keepalives on the tunnel interface detect tunnel-level failure. Layer 7 health checks (Cisco IP SLA, Juniper RPM) against http://gateway.<cloud>.net/vpntest detect service failures beyond the tunnel.

Do not use google.com as an L7 health-check target Using www.google.com as the health-check destination causes Google to rate-limit or CAPTCHA ZIA egress IPs. Use Zscaler’s own /vpntest endpoint.

Failover timing trade-offs: fast failover (30–42 seconds worst case; higher false-positive risk) versus conservative (120–185 seconds worst case; lower false-positive risk, dependent on L7 health test).

Deployment scenarios

Internal router (preferred)
GRE tunnel from a router behind the firewall. Firewall allows GRE outbound. Preserves inner IP visibility for sublocation matching and per-user logging.
Border router
GRE from the border router. Must disable NAT on the firewall for inner IPs to remain visible to Zscaler; may conflict with other firewall policies.
Explicit proxy via GRE (no default route)
For sites with no default internet route that must proxy through Zscaler. Uses Global PSE IPs rather than geolocation-based VIP selection.

§ 3 · IPSec tunnels

IPSec tunnels

Encrypted branch tunnel — adds authentication and encryption overhead vs GRE in exchange for security over untrusted paths and support for dynamic-IP sites.

When to prefer IPSec over GRE

  • The path to Zscaler traverses the public internet and traffic must be encrypted in flight.
  • The branch has a dynamic IP — the UFQDN credential type supports dynamic IPs without requiring a static registration.
  • The branch is smaller and throughput requirements do not justify managing multiple GRE tunnels.
  • The router or firewall supports IKE/IPSec natively but not GRE.

VPN credential object

IPSec in ZIA uses a VPN credentials object (TrafficVPNCredentials) that is provisioned independently of the Location and then attached to it. The credential type determines how the tunnel peer is identified.

TypeIdentifierIP requirementNotes
IPSource IP addressStatic IP requiredStandard IPSec branch tunnel; IP is the IKE peer identifier.
UFQDNFQDN string (e.g., branch@corp.example.com)No static IP requiredDynamic-IP branch tunnel; FQDN is the IKE identifier.
CNX.509 Common NameCertificate-basedLegacy or cert-based deployments. Less common.
XAUTHXAUTH challenge identifier—Legacy; pre-IKEv2 deployments.

IP and UFQDN are the standard production types. Key SDK constraints: type and fqdn/ip_address are immutable after creation — changing either requires deleting and recreating the credential. The pre_shared_key is write-only (sensitive) and not returned on read. Bulk-delete is capped at 100 credential IDs per call.

MTU considerations

IPSec ESP adds approximately 50–80 bytes of overhead (more than GRE’s 4 bytes), depending on cipher suite and whether NAT-T (UDP port 4500) is in use. On a 1,500-byte WAN MTU, effective IPSec payload MTU is typically 1,380–1,420 bytes. MSS clamping on the IPSec tunnel interface is as critical as on GRE — failure causes large-transfer problems more commonly under IPSec because the overhead is higher.

Source gap No Zscaler-authored IPSec configuration capture covers specific IKE phase parameters, supported cipher suites, or NAT-T behavior in the vendored source material. Treat exact cipher requirements as unverified until cross-checked against the live help portal. ZIA PSEs accept IKEv1 and IKEv2; IKEv2 is preferred.

API and Terraform

  • GET/POST/PUT /zia/api/v1/vpnCredentials — credential CRUD
  • POST /zia/api/v1/vpnCredentials/bulkDelete — bulk delete (max 100 IDs)
  • Terraform: resource_zia_traffic_forwarding_vpn_credentials
§ 4 · PAC files

PAC files

A PAC file is a JavaScript program (FindProxyForURL) that tells a browser where to send each request — directly or through a named proxy. The browser fetches the PAC URL, executes it per request, and follows the returned directive.

When PAC is the right choice

  • Tunnel provisioning is not possible (cloud-hosted, SaaS-only environments, no VPN-capable router).
  • BYOD or guest devices that cannot install an agent.
  • Only web and browser traffic (HTTP/HTTPS) needs inspection — PAC intercepts browser traffic only.
  • ZCC is not deployed and GRE/IPSec tunnels are not available.

Zscaler-hosted PAC files

Zscaler generates and hosts four master PAC files. They are non-editable — to customize, clone and upload as a custom PAC.

PACPurposePort
recommended.pacGeneral browser forwarding. Default choice for new deployments.80
proxy.pacLegacy/simpler variant.80
mobile_proxy.pacMobile device forwarding.80
kerberos.pacFor tenants using Kerberos authentication — forces FQDN addressing via ${GATEWAY_HOST} so SPNs resolve correctly.8800

Any tenant that enables Kerberos must use kerberos.pac (or a clone), even when traffic is also forwarded via IPSec or GRE.

Server-side variable substitution

When a browser fetches a Zscaler-hosted PAC URL, Zscaler’s PAC server runs geolocation against the client’s source IP to find the nearest PSE, then performs server-side variable substitution — inserting current PSE IPs and FQDNs into the file before returning it. This is the key mechanic: the PAC body is rendered per-request with tenant-specific and location-specific values.

VariableSubstituted with
${GATEWAY}Primary PSE IP address
${SECONDARY_GATEWAY}Secondary PSE IP address
${GATEWAY_HOST}Primary PSE FQDN (required by Kerberos)
${SECONDARY_GATEWAY_HOST}Secondary PSE FQDN
${GATEWAY_FX}Primary PSE IP for manual-failover templates
${SECONDARY_GATEWAY_FX}Secondary PSE IP, manual-failover variant
${GATEWAY.<subcloud>.<cloud>.net}Subcloud-scoped primary PSE
${SECONDARY.GATEWAY.<subcloud>.<cloud>.net}Subcloud-scoped secondary PSE
Self-hosting loses variable substitution Variable substitution only works when the PAC is served from Zscaler’s cloud. Copying a PAC to an internal web server causes ${GATEWAY} to become a literal string. Self-hosted PACs lose the geolocation and automatic-failover benefits.

WPAD / auto-discovery

Browsers and operating systems can auto-discover the PAC URL via WPAD (Web Proxy Auto-Discovery). Two delivery mechanisms:

  • DHCP option 252 — the DHCP server returns the PAC URL in option 252. The client uses it without DNS lookup. Preferred on managed networks.
  • DNS WPAD — the client resolves wpad.<domain> and fetches http://wpad.<domain>/wpad.dat. Requires DNS and web server configuration. Less reliable across domain boundaries.

Both mechanisms rely on the client’s proxy auto-detection setting being enabled. Managed Windows devices via Group Policy should receive the PAC URL via GPO rather than WPAD for reliability.

Version states: DEPLOYED, STAGE, LKG (Last Known Good). SDK supports: list, get, add, clone, validate, update (with action DEPLOY / STAGE / LKG / UNSTAGE / REMOVE_LKG), delete. The add_pac_file() call validates syntax before upload; standalone validation is available via validate_pac_file().

PAC changes are immediate There is no activation gate. The moment a PAC version is deployed, all browsers fetching that URL receive new content on their next refresh. Browsers cache PAC files for minutes to hours — full rollout can take up to an hour. Apply changes in a maintenance window. No Terraform resource exists for PAC file management in the vendored provider; use the SDK or Admin Portal.

Common PAC logic patterns

function FindProxyForURL(url, host) {

  // Bypass intranet and local addresses
  if (isPlainHostName(host) ||
      shExpMatch(host, "*.corp.example.com") ||
      isInNet(host, "10.0.0.0", "255.0.0.0")) {
    return "DIRECT";
  }

  // Bypass KDC proxy to prevent Kerberos re-challenge loops
  if (shExpMatch(host, "kdcproxy.corp.example.com")) {
    return "DIRECT";
  }

  // Route all other traffic through ZIA PSE (primary + failover)
  return "PROXY ${GATEWAY}:80; PROXY ${SECONDARY_GATEWAY}:80; DIRECT";
}

Performance note: PAC files execute serially on every request. Avoid dnsResolve() and isResolvable() except where required — DNS inside a PAC is a synchronous blocking call. Group large bypass lists under a single outer if and put high-probability checks at the top.

Limitations

Browser-only
PAC only intercepts traffic the browser or OS proxy stack handles. Non-browser apps (system services, thick clients, ICMP, UDP services) bypass PAC entirely.
No UDP
Explicit proxy is TCP-only. QUIC, RTP, DNS-over-UDP, and other UDP protocols are invisible to PAC-forwarded sessions.
No non-HTTP protocols
FTP, SSH, SMTP, and similar protocols are not redirected by PAC.
Authentication challenges
When Enforce Authentication is on for the Location, browsers receive a 407 Proxy Authentication Required. Domain-joined Windows browsers handle NTLM/Kerberos proxy auth transparently; non-domain or non-browser HTTP clients may fail silently or loop. ZCC eliminates this for managed devices.
Location binding not required
PAC-forwarded traffic can reach ZIA without a pre-configured Location if the user is authenticated. Without a Location, location-scoped policy doesn’t apply and reporting granularity is reduced.
§ 5 · Explicit proxy mode

Explicit proxy mode

When traffic arrives at the ZIA PSE with the PSE’s own IP as the destination — via PAC file or ZCC Z-Tunnel 1.0 — the PSE operates in explicit proxy mode. The client sends an HTTP CONNECT request naming the real destination before the SSL handshake begins.

Explicit vs transparent

ModeArrives viaHow PSE identifies destination
ExplicitPAC file, ZCC Z-Tunnel 1.0, proxy chainingHTTP CONNECT host header — plaintext, before TLS, unambiguous
TransparentGRE tunnel, IPSec tunnel, ZCC Z-Tunnel 2.0TLS SNI extension in Client Hello; fallback to server certificate if SNI absent

Proxy mode is a consequence of the forwarding method — it is not configured directly. The choice of forwarding method determines which mode applies.

Authentication challenge behavior

In explicit proxy mode, when Enforce Authentication is enabled on the Location, the PSE issues a 407 Proxy Authentication Required response. This fires before any content inspection. Modern browsers (Chrome, Firefox, Edge) handle NTLM and Kerberos proxy auth natively on domain-joined machines. Off-domain browsers prompt interactively. Non-browser HTTP clients vary — some fail silently on 407, others enter a retry loop.

Under transparent (GRE/IPSec) mode, authentication challenges work differently — ZIA injects a redirect to an authentication portal rather than issuing a 407. This is less disruptive to non-browser traffic but requires the client to follow the HTTP redirect.

SSL inspection implications

In explicit mode, the PSE has a reliable destination hostname from the CONNECT request before any TLS negotiation. SSL Inspection policy, URL Filtering, and Cloud App Control all evaluate against this CONNECT host at the pre-decrypt pass. This eliminates the “Miscellaneous or Unknown” over-exemption risk that exists under transparent forwarding, where SSL rules can match on destination IP categories for traffic lacking an SNI extension.

CONNECT User-Agent criteria in SSL Inspection rules only fire in explicit proxy mode — transparent-proxy traffic (GRE/IPSec/Z-Tunnel 2.0) never carries a CONNECT User-Agent, so those rule criteria silently do nothing for tunnel-forwarded traffic.

Ports

Standard explicit proxy ports: 80 (HTTP) and 443 (HTTPS). Kerberos tenants use port 8800 for authentication challenges. Zscaler also supports dedicated proxy ports per Location for multi-tenant or partner configurations.

§ 6 · Source IP anchoring

Source IP anchoring (SIPA)

A ZIA + ZPA cross-product feature that forwards ZIA-processed traffic through a customer-deployed ZPA App Connector before delivery to the destination, so the destination sees the App Connector’s IP — not Zscaler’s shared cloud egress IPs.

Why SIPA exists

In a standard Zscaler deployment, traffic egresses from Zscaler’s Public Service Edge IPs — shared infrastructure across many tenants. This breaks access when the destination enforces a static IP allowlist, a compliance requirement that traffic originate from corporate-owned IP ranges, or source-IP-based conditional access (most notably Microsoft 365 Conditional Access).

SIPA routes the last hop through a ZPA App Connector inside the customer’s network or in a chosen datacenter. The destination sees the App Connector’s IP as the source. ZIA inspection still runs first — SIPA changes the egress IP, not the inspection pipeline.

M365 Conditional Access scope For Microsoft 365, SIPA only needs to cover the initial login traffic (login.microsoftonline.com, login.windows.net, login.microsoft.com). Post-authentication app traffic uses tokens; source-IP validation no longer applies. Scope the SIPA App Segment narrowly to minimize App Connector load.

Traffic flow

Normal ZIA:
Client → ZIA PSE → destination
                 (destination sees Zscaler PSE IP)

SIPA:
Client → ZIA PSE → ZPA Gateway → ZPA App Connector → destination
                                              (destination sees App Connector IP)

The full ZIA inspection pipeline (URL Filter, SSL Inspection, DLP, CAC, Sandbox, Malware, ATP, Firewall, IPS) runs at the ZIA PSE before the forwarding decision. ZPA is the egress path, not an inspection bypass.

Configuration overview

SIPA requires coordinated configuration in both the ZIA and ZPA Admin Portals. A full ZPA license is not required — a SIPA subscription plus App Connectors deployed for this purpose is sufficient.

ZPA side

  • Create an App Segment with Source IP Anchor enabled and Bypass set to Use Client Forwarding Policy.
  • Client Forwarding Policy: for domain-based apps, two rules — Bypass ZPA for all client types except ZIA Service Edge, and Forward to ZPA for ZIA Service Edge only. For IP-based apps, a single Only Forward Allowed Applications rule.
  • Access Policy: allow access for the ZIA Service Edge client type. Do not add user-based SAML/SCIM criteria — ZIA handles user scoping; duplicating it in ZPA breaks SIPA in a hard-to-diagnose way because ZPA sees ZIA Service Edge as the client identity, not the original user.

ZIA side

  • Configure a ZPA Gateway at Administration > ZPA Gateway referencing the ZPA Server Group and App Segments by their ZPA-side external IDs.
  • Create a Forwarding Control rule with Forwarding Method = ZPA scoped to the SIPA App Segments and the ZPA Gateway.
  • Enable both predefined DNS Control rules: ZPA Resolver for Road Warrior (higher precedence) and ZPA Resolver for Locations. Road Warrior must have higher rule precedence than Locations — if reversed, road-warrior traffic is resolved through the wrong IP pool.
Z-Tunnel 1.0 / PAC gating To support SIPA for Z-Tunnel 1.0 or PAC road-warrior traffic, Enable Firewall for Z-Tunnel 1.0 and PAC Road Warriors must be on in Advanced Settings. Without this toggle, Z-Tunnel 1.0 and PAC traffic bypasses Forwarding Control evaluation entirely.

SIPA Direct — disaster recovery variant

When ZIA is in disaster recovery (unreachable or impaired), the normal SIPA flow breaks because ZIA Service Edge is unavailable to initiate the forward-to-ZPA leg. SIPA Direct reconfigures the ZPA Client Forwarding Policy to route ZCC clients directly through ZPA connectors, bypassing ZIA temporarily.

SIPA Direct must be reverted manually when ZIA recovers — there is no automatic reversion. The Access Policy must include an explicit rule allowing ZCC client access to SIPA apps; otherwise Direct mode fails during the DR window. Pre-plan this before a DR event occurs.

Protocol limitations and mutual exclusions

SIPA does not support RTSP. ICMP echo request/response is supported on ICMP-enabled App Segments with a 990-byte payload cap; ICMP traceroute is not supported.

A SIPA-enabled App Segment cannot simultaneously use Browser Access, Double Encryption, or Multimatch (match_style = INCLUSIVE). These mutual exclusions are validated at segment-save time.

The ZIA SSL Inspection rule’s zpa_app_segments selector only lists App Segments with Source IP Anchor enabled. Non-SIPA segments are invisible to that SSL rule criterion.

§ 7 · Forwarding control policy

Forwarding control policy

Forwarding Control is the ZIA policy module that decides where traffic goes after all content inspection has run. It is not an on-ramp — it is the egress routing decision at the far end of the ZIA pipeline.

Pipeline position

Traffic arrives at ZIA PSE (via GRE / IPSec / PAC / ZCC)
      ↓
Firewall Control
      ↓
SSL Inspection (decrypt or bypass decision)
      ↓
URL Filtering / CAC / DLP / Sandbox / Malware / ATP
      ↓
[Forwarding Control]   ← you are here
      ↓
Egress: internet / ZPA App Connector / proxy chain / drop

Because Forwarding Control fires after content inspection, a ZPA-forwarded flow still traverses DLP, SSL Inspection, and IPS. Forwarding Control selects the exit ramp; it does not bypass the inspection pipeline that precedes it.

Forward methods

forward_methodBehavior
ZIADefault egress via Zscaler’s shared cloud egress IPs. No special config required. Applied when no rule matches.
ZPARoute via ZPA App Connector for SIPA. Requires zpa_gateway + zpa_app_segments.
ECZPALike ZPA but for Zscaler Cloud Connector (Edge Connector) deployments.
DIRECTBypass ZIA’s egress infrastructure; send traffic directly to the destination.
PROXYCHAINForward to a downstream proxy. Requires proxy_gateway.
DROPSilently discard matching traffic.
ENATDEDIPEgress through a dedicated IP gateway (customer-attributable egress IP, Zscaler-provisioned infrastructure). Separate ZIA SKU required.
GEOIPRoute based on destination geo-IP classification.

dest_addresses, dest_countries, and dest_ip_categories can only be set on DIRECT, PROXYCHAIN, ENATDEDIP, and GEOIP rules — the Terraform provider enforces this at plan time. ZPA rules scope by zpa_app_segments instead.

Predefined rules

Four predefined forwarding rules ship with every ZIA tenant and cannot be deleted:

Rule namePurpose
ZIA Inspected ZPA AppsForwards ZPA app-segment traffic with “Inspect Traffic with ZIA” enabled back through ZIA for inspection. Requires ZPA App Inspection license.
Fallback mode of ZPA ForwardingFallback handling when the primary ZPA forwarding path is unavailable.
Client Connector Traffic DirectApplies direct forwarding for ZCC-originated traffic matching defined criteria.
ZPA Pool For Stray TrafficCatches ZPA-destined traffic that doesn’t match any configured forwarding rule.

Rule criteria

Forwarding Control rules share the standard ZIA rule-criteria model. Identity scoping: users, groups, departments, device_groups. Location scoping: locations (up to 32), location_groups (up to 32). Source: src_ips, src_ip_groups, src_ipv6_groups. Destination: dest_addresses, dest_ip_groups, dest_ip_categories, dest_countries. Network: nw_services, nw_service_groups, nw_application_groups. App: zpa_app_segments (for ZPA method), app_service_groups.

Rule evaluation is first-match-wins in ascending rule order, with Admin Rank as a structural gate — same model as URL Filtering and Firewall.

ENATDEDIP vs SIPA

Both solve “I need a stable, customer-attributable egress IP” but differ on infrastructure ownership. SIPA routes through a customer-deployed ZPA App Connector (customer VPC or datacenter IP). ENATDEDIP routes through a Zscaler-provisioned dedicated egress gateway with a dedicated public IP. The ENATDEDIP forward method requires a separate Dedicated IP ZIA SKU; the API returns INVALID_INPUT_ARGUMENT with a “Dedicated IP Gateway is required” message if the tenant is unlicensed.

§ 8 · Subclouds

Subclouds

A subcloud restricts which PSEs serve a tenant’s traffic. Default behavior uses geolocation — subclouds override this to keep traffic inside a specified regional footprint for data-residency or compliance requirements.

What a subcloud is

A subcloud can be: a subset of Public Service Edges only (e.g., EU-region PSEs for GDPR); a subset of Private Service Edges only; or a mix of Public and Private Service Edges. Subclouds cannot consist of PSEs in a single datacenter — a minimum of two datacenters is required for availability.

Subclouds are not self-service. They require a Zscaler Support ticket and are only provisioned for organizations with access to limited, restricted, or private datacenters.

How clients route to a subcloud

Unqualified ${GATEWAY} resolves via geolocation — it may or may not respect the subcloud depending on how the tenant is configured. Use the subcloud-qualified variable form in PAC files to pin traffic to the correct PSE pool.

${GATEWAY.<Subcloud>.<Zscaler cloud>}
${SECONDARY.GATEWAY.<Subcloud>.<Zscaler cloud>}

// Kerberos (FQDN form):
${GATEWAY.<Subcloud>.<Zscaler cloud>_HOST}

// Example: US-only CONUS subcloud (Zscaler-managed, no ticket needed):
${GATEWAY.CONUS.zscaler.net}

The subcloud identifier in the variable must match the exact DNS name assigned to the subcloud in the Admin Portal (Administration > Resources > Subclouds). Check the UI for the exact string.

Propagation timing

LayerRefresh interval
Zscaler hosted PAC files~5 minutes
ZCC Application Profile PAC15 minutes
Effective user redirect10–20 minutes

Propagation is cascaded, not atomic. If a client is forced to refresh policy before the 5-minute Hosted PAC mark, it pulls a stale config. Wait for the 5-minute mark before triggering a client policy update. During the staggered window, some clients are on the new PSE set and others are on the old.

Business Continuity Cloud bypasses subclouds When BC Cloud activates during a PSE outage, traffic routes through Zscaler’s BC infrastructure regardless of subcloud config. BC Cloud supports only Z-Tunnel 1.0, PAC, and GRE — tenants relying on Z-Tunnel 2.0 plus a restrictive subcloud lose both capabilities during BC activation.

Country / tz read-write asymmetry (ZIA API, mid-2026) The ZIA API’s GET /locations response returns country = "NETHERLANDS" and tz = "NETHERLANDS_EUROPE_AMSTERDAM" (unprefixed). TF v4.7.18 removed the validator entirely to align with the API change. Whether the ZIA API write-side now requires prefixed, unprefixed, or accepts both is not confirmed in source material — treat write-side current state as tier-C until lab-verified. See zia-fwd-01.

§ 10 · Triage table

Triage table

Symptom-first lookup for common forwarding failures. Start here when diagnosing branch connectivity issues.

Symptom Most likely cause Reference
“Traffic not reaching ZIA through GRE.” Source IP not registered as a TrafficStaticIP, or registered IP changed (dynamic IP used instead of static). Location binding fails silently — traffic bypasses ZIA. §2 Static IP
“GRE tunnel up but large file transfers hang or stall.” MTU/MSS mismatch. GRE encapsulation adds 24 bytes (IP+GRE). MSS not clamped to 1436 on the tunnel interface. §2 MTU / MSS
“GRE throughput caps at ~250 Mbps even though the WAN link is 1 Gbps.” Inner endpoint IPs are NATed. All traffic appears from one source IP; load-balancer cannot spread across nodes. 250 Mbps is the NATed ceiling. §2 Bandwidth
“Browser ignoring PAC file.” WPAD auto-detection disabled in browser or OS settings; PAC URL not pushed via GPO or DHCP option 252; PAC file syntax error (use Admin Portal Verify or pactester). §4 WPAD, §4 Custom PAC
“PAC file changes not taking effect across all users.” Browser PAC caching — browsers cache the PAC for minutes to hours regardless of server-side change. Full propagation can take up to an hour. §4 Custom PAC
“Authentication prompt appearing for GRE traffic.” auth_required toggled on for the Location or a sublocation. GRE-forwarded traffic under transparent mode sees a redirect-to-auth-portal, not a 407. If the prompt is a 407, the traffic is actually arriving via explicit proxy (PAC/ZCC Z-Tunnel 1.0), not GRE. §5 Auth challenge
“Repeated 407 prompts for browser using PAC.” Browser is not domain-joined or not passing Kerberos/NTLM proxy auth. Non-browser HTTP clients (Curl, app HTTP libraries) often fail silently on 407 rather than prompting. Deploy ZCC on managed devices to handle auth internally. §4 Limitations
“Wrong ZIA node (datacenter) selected for GRE tunnel.” within_country not set, causing VIP selection to use geolocation rather than in-country DCs. Or: primary_dest_vip was set to a surcharge/premium DC, which bypasses the normal picker. Check the configured VIP against GET /zia/api/v1/vips/recommendedList. §2 VIP selection
“Sub-location traffic getting parent location policy instead of sublocation policy.” Traffic inner IP does not match any defined sublocation range — it falls into the other catch-all sublocation. Check IP range definitions and verify xff_forward_enabled if relying on XFF for sublocation matching. §1 Sublocations
“New location not inheriting expected policy rules.” location_type set to the wrong value — predefined dynamic group enrollment happens at write time. Or: rules scope by a manual Location Group that the new Location was never added to. §1 Location Groups
“SSL inspection rule with CONNECT User-Agent criterion not firing for GRE traffic.” CONNECT User-Agent criteria only fire in explicit proxy mode (PAC / Z-Tunnel 1.0). GRE traffic arrives in transparent mode — there is no CONNECT request, so the criterion can never match. §5 SSL implications
“SIPA traffic exiting with Zscaler PSE IP instead of App Connector IP.” Multiple possible causes in sequence: (1) ZPA App Segment does not have Source IP Anchor enabled; (2) ZPA Client Forwarding Policy uses wrong action type for domain-based vs IP-based apps; (3) ZPA Access Policy blocks the ZIA Service Edge client type; (4) ZIA Forwarding Control rule not set to ZPA method; (5) ZIA DNS Control rules for ZPA Resolver not enabled or in wrong order. §6 Config
“IPSec credential update fails — ‘type is immutable.’” type and fqdn/ip_address on a VPN credential are immutable after creation. Delete the credential and recreate it with the new values. §3 VPN credential
“PAC file returns wrong PSE (not in EU despite EU subcloud).” PAC uses unqualified ${GATEWAY} instead of ${GATEWAY.<Subcloud>.<cloud>.net}. Unqualified variable resolves by geolocation, ignoring the subcloud. §8 Subcloud routing
“Terraform plan fails with country / tz validation error on Location resource.” Using TF provider pre-v4.7.18 with the API’s new unprefixed country/tz return values. Upgrade to v4.7.18+ or normalize values to the prefixed form expected by the old schema validator. §9 Location fields
“ZCC Z-Tunnel 2.0 on-network users unexpectedly operating in Z-Tunnel 1.0 mode.” Z-Tunnel 2.0 requires a single NAT IP for all connections from a given device. ECMP or multiple GRE egress IPs split control and data connections across different Service Edges, causing Z-Tunnel 2.0 to fall back silently to Z-Tunnel 1.0. §2 Bandwidth (multiple source IPs note)
