02 — What's in it 03 / 06

Four pillars. Both audiences.

Engineers consume directly — read references, follow methodology docs, run scripts. Agents consume the same content via tool calls. Same files. Same single source of truth.

The pinned bibliographyKnowledge
~220 curated reference files plus vendored upstream sources (SDK, Terraform, Postman, help docs) as pinned bibliography. Engineers browse via the docs site or directly in the repo. Agents cite the same files in their discovery journals.
How work actually happensDiscipline
Methodology docs for investigation / audit / architect — citation-anchored, status-enumed, verification-gated. Engineers read them as work guidance. Agents follow them as procedure.
Structured proceduresWorkflows
Three slash commands — /z-investigate, /z-audit, /z-architect — emit registers (journal / findings / recommendations). Primarily an agent surface, but the playbook content is reference-grade for any engineer wanting to understand the methodology.
Operator-grade qualityTooling
Five hygiene checks in CI. Eval suite of 19 behavioral specs. Vendor-drift detection. Scrape freshness. The pipeline ensures the kit is reliable for both human readers and agent reasoning — same checks, same standards.

05 — Where it's headed 06 / 06

Roadmap & what's next.

Tracked in IMPROVEMENTS.md. The kit gets stronger every cycle — the discipline compounds.

Validates the templateFirst production-verified bundle
When a real investigation or scaling review lands, capture it as the canonical bundle. Use it to confirm the template format works against real evidence — not to manufacture speculative content.
Cross-agent determinismSchema-validated structured output
Registers (discovery journal, audit findings, recommendations) currently emit Markdown. Pair with a JSON sidecar that scripts can validate. Same content, machine-checkable.
When subtype #2 landsSubtype-parameterize /z-audit
Today's audit is the lint flavor. Policy, access, coverage, and config subtypes are sketched. Activate the second one when there's a real use case — auto-detect-from-scope vs. explicit subtype arg.
Two questions worth askingWhat's next for each audience?
For agents: each new /z-* command bottles operator expertise. What's the next workflow worth scaffolding? For engineers: which knowledge gap should the references/ side close first — what do you find yourself re-explaining most?
