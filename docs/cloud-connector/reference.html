At a glance

  • Not ZCCCloud Connector is a purpose-built VM, not an endpoint agent. It forwards workload traffic from EC2 instances, containers, and servers. ZCC is for user devices. Never install ZCC on a workload and call it done.
  • VM in your cloudThe CC instance runs inside the customer's AWS, Azure, or GCP account. Outbound-only — it dials out to Zscaler's Zero Trust Exchange; no inbound connection from Zscaler ever reaches it.
  • Workload → CC → ZTEWorkload traffic hits the Cloud Connector (via GWLB or route-table redirect), which forwards it to ZIA (internet-bound) or ZPA (private-app-bound). ZIA policy applies as if traffic originated at a ZIA location.
  • Five methodsEach forwarding rule assigns one of: ZIA, ZPA, Direct, Drop, or Local. First-match-wins, top-down rule order. A terminal default ZIA rule exists in every group.
  • Outbound onlySame structural property as ZPA App Connectors. No public listener to protect. The CC's data plane connects to ZIA/ZPA Service Edges over outbound TLS tunnels.
  • Group = policy unitCloud Connector Groups are the unit for policy assignment, autoscaling scope, and upgrade scheduling. A provisioning template assigns a VM to a group at boot.
  • Activation gateConfiguration changes are staged-pending until activated, parallel to ZIA's model. Not ZPA-style (which propagates on write). Use POST /ztw/api/v1/ecAdminActivateStatus/activate.
  • SDK namingThe product has five names: Cloud Connector, Branch Connector, Zero Trust Gateway, ZTW (Go SDK module), ZTC (Terraform provider). All refer to the same product family. Python SDK ZTW coverage landed in v2.0.0 (2026-04-22).
§ 0 · Deployment model

Deployment model

A Cloud Connector instance is a virtual appliance running in the customer's cloud account, grouped for policy and scaling, deployed via a provisioning URL that embeds tenant identity at boot.

Fig. 1Workload traffic is transparently redirected to the Cloud Connector, which tunnels it to the Zero Trust Exchange. ZIA inspection applies before the traffic reaches the internet.

Cloud Connector Groups

Every Cloud Connector VM belongs to exactly one Cloud Connector Group, assigned at boot via the provisioning template. A group is the unit of:

  • Policy application — forwarding rules reference groups as a match criterion.
  • Upgrade orchestration — the weekly upgrade window is set per group; upgrades stagger across member VMs.
  • Autoscaling scope — an ASG (AWS), VMSS (Azure), or MIG (GCP) maps one-to-one to one Cloud Connector Group.

Group types

The admin console Group Type dropdown exposes two values: Cloud Connector (traditional) and Zero Trust Gateway (ZTG) (a newer variant). The functional distinction between the two is not fully documented in available sources. Both appear in production tenants; the difference is likely an architectural naming evolution.

Per-VM states

Active
Healthy; processing workload traffic.
Inactive
Enrolled but not currently processing traffic.
Disabled
Admin-disabled; traffic processing stops but the VM stays in inventory. Useful for staged rollouts or incident response without deprovisioning.

Template mismatch constraint

Autoscaling group (ASG/VMSS/MIG) deployments and non-autoscaling deployments each require their own provisioning template. Deploying an ASG CloudFormation template into a VPC that already has a non-ASG deployment is explicitly documented as unsupported.

HA and failover

All Cloud Connectors in a group are active simultaneously — no active/passive. A cloud-provider native load balancer distributes traffic across them. Health is validated via HTTP probe to /cchealth on each CC instance; 200 = healthy, 503 or timeout = unhealthy.

Gateway failover chain

Each CC Group is configured with a primary ZIA/ZPA gateway. If the primary is unreachable, CC fails over to a secondary in approximately 30 seconds. A tertiary gateway is automatically tried if both fail; it is not user-configured. When the primary recovers, new sessions route back to it — existing sessions on the secondary continue until they terminate naturally.

Fail-close vs fail-open

ModeIf all gateways unreachable
Fail-close (default)Traffic dropped. Workload cannot reach internet until ZTE connectivity is restored.
Fail-openTraffic egresses direct, bypassing Zscaler inspection. Availability over security.
Semantics are inverted from common expectation "Fail-open" in Cloud Connector means internet traffic egresses uninspected — not that access is blocked. "Fail-close" drops traffic, not that it allows. Read these terms carefully when configuring.

Health check customization

Default health check intervals are 30 seconds (AWS GWLB) and 15 seconds (Azure ILB). Changing these requires Zscaler Support engagement.

Existing sessions during failover

The 30-second gateway failover time applies to new-session routing. Existing sessions on a failed CC or gateway may time out and require the workload to retry. Applications with long-lived connections — database connections, streaming protocols — are affected.

Provisioning flow

The ZIA Admin Console generates a per-template provisioning URL that bootstraps each CC instance at boot. The URL carries tenant identity, group assignment, location, and VM-size configuration.

Create a provisioning template.Navigate to Infrastructure → Connectors → Cloud → Management → Provisioning. Select cloud provider (AWS / Azure / GCP), location template, group settings, and VM size.
Save to generate the provisioning URL.The admin portal emits a prov_url. In Terraform, ztc_provisioning_url creates this programmatically with cloud_provider_type = "AWS", "AZURE", or "GCP".
Inject the URL into the VM at launch.Passed via EC2 user_data (AWS) or Azure cloud_init / custom_data. For ASG deployments, it lives in the Launch Template so scale-out events inherit it automatically.
CC reads credentials from cloud-provider secret store.AWS: Secrets Manager. Azure: Key Vault via Managed Identity. The CC AMI/image reads credentials at boot to complete ZIA registration.
Activate pending config.After any config changes, run POST /ztw/api/v1/ecAdminActivateStatus/activate or apply ztc_activation_status in Terraform.
Stale provisioning URL in Launch Template If the ZIA provisioning URL is regenerated (key rotation, template recreation) and the Launch Template or VMSS model is not updated, scale-out events produce CCs that fail to register with ZIA. This failure presents as a new CC that never appears in the admin console inventory.
§ 1 · Forwarding mechanics

Forwarding mechanics

Traffic forwarding rules are the policy engine. Rules match workload traffic on network and application criteria and assign one of five forwarding methods. Evaluation is first-match-wins, top-down.

The five forwarding methods

MethodWhat it doesCommon use
ZIAForward to ZIA for internet inspection. ZIA policy applies as if traffic came from a ZIA location.Default for all internet-bound workload traffic. A pre-provisioned terminal rule with method ZIA exists in every group.
ZPAForward to ZPA for private-app access through the zero-trust tunnel.Workload-to-workload and workload-to-datacenter. Requires CC ZPA enrollment, a matching ZPA Application Segment, and Access Policy permit.
DirectBypass Zscaler entirely; route via cloud provider's native networking.Cloud metadata endpoints (169.254.169.254), intra-VPC health checks, VPC-private storage endpoints. Traffic is uninspected.
DropDiscard. Workload sees connection failure.Explicit deny — block specific destinations for workloads.
LocalForward locally within the CC's own network context. Cloud Connector and ZTG only; not available on Branch Connector.Cloud-to-cloud workload communication where local inspection applies.

ZPA forwarding failure modes

When a ZPA rule matches but traffic fails, the common causes in order of frequency: (1) no matching ZPA Application Segment exists for the destination, (2) ZPA Access Policy denies the Cloud Connector's identity, (3) App Connector unreachable. Diagnose ZPA-side first before suspecting CC.

Predefined rules

Two predefined rules ship in every CC group and are gateway-mode-only, license-gated: a Direct for Zscaler Cloud Endpoints rule (routes CC's own control-plane traffic direct) and a WAN/LAN Destinations Group rule. Non-gateway-mode deployments and tenants without the relevant license tier will not see them.

Rule structure and evaluation

Each rule has a Forwarding Rule section (order, name, status, method) and a Criteria section. All criteria are ANDed across fields; multiple values within a single field are ORed — the same AND-across-fields / OR-within-field pattern as ZIA URL Filter.

Key criteria fields

FieldLimitNotes
Location / SublocationUp to 8Empty = applies to all locations.
Cloud & Branch Connector GroupsUp to 32Empty = applies to all groups.
Network Services / GroupsNo stated limitCustom and predefined groups.
Application Service Groups—Zscaler-maintained sets: Office365, Zoom, Webex, AWS, Azure, GCP, Zscaler Cloud Endpoints, others. Updated by Zscaler — new endpoints added automatically affect matching rules.
Source IP Groups—Workload-side source IPs.
Destination IP Groups—Destination addresses.
Domains / FQDN—Domain-based matching. Requires DNS through CC for non-HTTP traffic (see §4 DNS subsystem).

Rule order and disabled rules

Rules evaluate in ascending numerical order. Disabled rules skip without losing their position in the evaluation sequence — the same pattern as ZIA URL Filter. The pre-provisioned default ZIA rule sits at the terminal position; traffic not matching any custom rule falls through to it.

First-match-wins footgun Placing a Direct exemption rule below a broad ZIA rule causes the ZIA rule to match first and the exemption never fires. Exemption rules for metadata services and cloud-provider storage endpoints must sit above the broad ZIA rule in rule order.

DNS dependency for FQDN matching

HTTP and HTTPS traffic can be matched by SNI/Host header even when DNS does not flow through CC. UDP and non-web TCP traffic to wildcard or FQDN destinations only matches if DNS flows through CC — without it, CC sees a destination IP, not a hostname, and FQDN criteria do not fire.

Log and Control Forwarding rules

A separate rule type, distinct from traffic forwarding rules, controls where CC sends its own logs, enrollment traffic, policy sync, and software update traffic. Operators debugging "Cloud Connector logs not reaching SIEM" should check Log and Control Forwarding Rules, not traffic forwarding rules.

Source IP anchoring behavior

When Cloud Connector forwards traffic to ZIA, the Zero Trust Exchange processes it as if the traffic originated at a ZIA location. The source IP visible to the ZIA Service Edge is the CC's egress IP — the NAT Gateway IP (AWS/Azure reference pattern) or the CC service interface IP.

ZIA policies that reference source IP or location will match against this CC-side IP, not the original workload IP. Country-code matching in ZPA Access Policy rules, when used in conjunction with Cloud Connector forwarding, evaluates the last NATed public IP (the CC or ZIA PSE IP), not the workload's original IP.

ZIA location assignmentTraffic from Cloud Connector appears to ZIA as coming from the location associated with the CC's provisioning template. Sublocations differentiate workload groups within a single location when workload discovery is configured.

§ 2 · AWS deployment

AWS deployment

Cloud Connector on AWS uses a dual-ENI model, deploys via CloudFormation or Terraform, and integrates with Gateway Load Balancer for multi-AZ HA. Route-table modification is a mandatory post-deploy step that CloudFormation does not automate.

GWLB vs ENI endpoint patterns

The fundamental AWS topology choice: how does workload traffic reach the CC instance?

PatternRoute targetUse case
GWLB (preferred)Gateway Load Balancer Endpoint (GWLBe) in workload VPCMulti-VPC, multi-AZ production. GWLB distributes across CC instances; VPC endpoints in spoke VPCs send traffic to the GWLB target group. Source IP preserved to CC.
ENI / route-tableNetwork Interface (CC service ENI ID)Single-CC or single-AZ low-scale. Simpler; no GWLB. Not HA — failed instance means route must be updated manually or by automation.

Reference architecture — Transit/Egress VPC

Zscaler's reference architecture places CC instances in a dedicated transit/egress VPC separate from workload VPCs. Spoke workload VPCs route default traffic to the GWLB endpoint via VPC peering and route-table entries. This reduces CC instance count versus per-spoke deployment.

Dual-ENI per instance

Each CC instance uses two ENIs in fixed roles:

ENIRoleIP Forwarding
Management ENIControl plane — SSH, registration, health reportingNot required
Service ENIData plane — workload traffic forwardingRequired
Route-table modification is not automated After CloudFormation stack creation, you must manually update the workload subnet's route table: for GWLB, 0.0.0.0/0 → Gateway Load Balancer Endpoint; for non-GWLB, 0.0.0.0/0 → Network Interface → CC service ENI. Forgetting this step leaves workload traffic on its default path (direct internet egress or blackhole). It is the most frequently cited deployment failure.

Deployment paths

Terraform — github.com/zscaler/terraform-aws-cloud-connector-modules. Zscaler's reference module; contents not vendored in this repo.

CloudFormation — Downloadable templates via the help portal in two flavors: ASG and non-ASG. These are not interchangeable within the same VPC — choose before initial deployment.

Auto Scaling Groups

CC instances in an ASG use custom CloudWatch CPU metrics (not the standard EC2 CPUUtilization) for scaling decisions. Each CC instance reports custom metrics at one-minute intervals; the ASG scaling policy aggregates them across all instances. The custom metrics reflect CC-application-level load (tunnel throughput, connection table pressure), not raw hypervisor CPU.

The ASG deployment supports warm pools (pre-initialized instances that reduce scale-out latency) and lifecycle hooks (allow a new CC to complete ZIA registration before the ASG marks it InService).

Manage lifecycle through the ASG, not EC2 Console Stopping or rebooting a CC that is part of an ASG from the EC2 Console may trigger termination and replacement by the ASG. Manage CC lifecycle through the ASG (desired count, scale policies) rather than instance-level EC2 operations.

Credentials — Secrets Manager

The CC AMI reads credentials from AWS Secrets Manager at boot. Storing credentials in Secrets Manager is a documented prerequisite, not an optional step. The EC2 instance profile attached to CC instances must include secretsmanager:GetSecretValue for the CC credentials secret. Missing this IAM permission produces a startup failure that presents as a network issue rather than an auth error.

AWS workload discovery

Workload discovery gives Zscaler read access to an AWS account so that policies can reference workload identity — EC2 tags, security group IDs, IAM instance profile ARNs, VPC IDs — rather than raw IP addresses. It is a separate integration from CC VM deployment; a CC can forward traffic without workload discovery configured.

Account trust model

Adding an AWS account to the Zscaler console registers it as a discovery target. Zscaler assumes an IAM role in the operator's account using a three-party cross-account AssumeRole:

Zscaler AWS account (Trusted Account ID)
  └─ Trusted Role
        └─ AssumeRole → Operator's AWS account
                          └─ Operator IAM Role
                               └─ Trust policy: principal = Zscaler Trusted Role
                                                condition: sts:ExternalId = External ID

The External ID is a per-account value Zscaler generates and displays in the console. It must appear in the Condition block of the operator's IAM trust policy. If regenerated, the IAM role must be updated or the AssumeRole call is rejected.

Permission states

StateMeaning
PendingAccount registered; IAM role not yet deployed or sync not yet triggered.
AllowedIAM role correctly deployed; Zscaler can assume it and discover tags.
DeniedAssumeRole rejected — role not deployed, trust policy wrong, External ID mismatch, or wrong Trusted Account ID.

The transition from Pending to Allowed/Denied is not automatic after role deployment. An operator must click Refresh on the account in the Zscaler console. The "Latest Sync" column reflects when Refresh was last triggered, not real-time sync status.

EventBridge integration

AWS EventBridge provides the real-time update path — Zscaler receives event notifications as workloads start, stop, or change tags, rather than polling. The operator creates EventBridge rules in their account routing relevant EC2/ECS/Lambda lifecycle events to the Zscaler EventBridge bus identified by the Event Bus Name field in the account config.

Namespace tagging for overlapping CIDRs

When multiple VPCs served by the same CC have overlapping CIDR blocks, Zscaler cannot determine which tag set applies to a given IP address. The zs:namespace VPC tag resolves this. Apply it to each VPC (not individual instances) with a unique value per namespace. Zscaler constructs a two-level mapping: VPC endpoint → namespace → (IP + namespace) → tag list.

The "No of Duplicates IP" counter in the account details region view is the primary signal that namespace tagging is needed. A non-zero count means tag data for those IPs is nondeterministic until namespaces are applied.

Sublocation scopes

Scope typeRequires workload discovery
VPC EndpointNo
VPCYes
AccountYes
NamespaceYes

VPC Endpoint scope is the only type usable without workload discovery. Authentication is not supported for workload-traffic sublocations. All sublocations within a location must use the same scoping strategy.

AWS Account Groups

Account Groups bundle multiple AWS accounts so that a single CC Group can receive tag data from all of them — the indirection needed in an AWS Organizations model where workloads span many accounts. An Account Group is associated with CC Groups at creation time; a CC Group not associated with any Account Group has no workload tag visibility, even if accounts are onboarded, and no error is surfaced.

VNet design and routing

The reference architecture places CC instances in a transit/egress hub VNet. Spoke workload VNets route default traffic to the CC ILB frontend IP via UDR and VNet peering.

BGP route propagation When CC coexists with an ExpressRoute or VPN Gateway in the same VNet, learned routes can override the static 0.0.0.0/0 UDR. Disable BGP route propagation on the workload route table and rely on the explicit UDR.

NSG requirements

Two NSGs are deployed, one per NIC. Management NSG allows inbound TCP 22 from VirtualNetwork (SSH from bastion), inbound ICMP, and outbound TCP 12002 to 199.168.148.101 (Zscaler Support tunnel, when enabled). Service NSG allows all inbound from VirtualNetwork and all outbound — the permissive VirtualNetwork rule implicitly covers Azure Load Balancer health probe traffic.

NAT Gateway

The reference Terraform deploys one NAT Gateway per CC service subnet for egress to ZIA Public Service Edges. The NAT Gateway's public IP becomes the egress source IP that ZIA sees. For multi-AZ deployments, Zscaler recommends one NAT Gateway per availability zone — Azure NAT Gateways are zone-specific, so a shared gateway across AZs is a single point of AZ failure for egress.

VMSS autoscaling

Enabling autoscaling on the provisioning template is gated by Zscaler Support — the help docs state: "To enable Auto Scaling, VMSS, or a MIG with autoscaling, contact Zscaler Support." Toggling VMSS without this Support enablement produces a misconfigured provisioning template.

Mandatory Azure Function App

VMSS deployments require an Azure Function App with two functions:

Health Monitoring
Runs every 60 seconds. Terminates instances reporting health 0 for 5 consecutive checks, or flapping (7/10 unhealthy). Controlled by terminate_unhealthy_instances (default true).
Resource Sync
Runs every 30 minutes. Reconciles ZIA Cloud Connector Group membership against the VMSS instance list. If a CC exists in the Group but not in the VMSS, it cleans up the orphan from the Group. This is the documented orphan-cleanup mechanism.
Flex Consumption plan regional availability The Azure Flex Consumption plan is not available in all Azure regions. When unavailable and VNet integration is required, upgrade to Elastic Premium (EP1). Skipping this produces a VMSS deployment with no orphan cleanup or unhealthy-instance termination — the VMSS scales correctly but dead CC entries accumulate in the CC Group.
§ 4 · DNS subsystem

DNS subsystem

Cloud Connector's DNS handling is three distinct subsystems with separate admin surfaces, rule sets, and gateway models. Failures that look like "DNS isn't working" frequently trace to one layer while another layer is fine.

SubsystemAdmin locationPrimary function
DNS GatewaysAdministration → Gateways → DNS Gateway tabDefine upstream resolver pairs with failover behavior.
DNS PoliciesForwarding → DNS PoliciesRule-based control over DNS requests and responses, including DoH.
Log & Control ForwardingForwarding → Log and Control ForwardingRoute CC operational messages (enrollment, policy sync, software updates, traffic logs) to Zscaler cloud.

DNS Gateways

DNS Gateways redirect DNS queries received by CC to specific upstream resolvers. Each gateway has a primary DNS server and an optional secondary. IPv6 resolvers are not supported.

Up to 255 DNS gateways can be created. A default DNS gateway exists and cannot be deleted.

Failure behavior options

Failure behaviorWhat happens when primary is unreachable
Return error responseCC returns SERVFAIL to the requesting client. Operationally safe but visible — clients get an explicit resolution failure.
Forward to Original DNS ServerDNS packet forwarded to the server the client was originally trying to reach. Preserves connectivity at the cost of bypassing whatever policy the gateway was meant to enforce.

Whether the secondary resolver is tried before the failure behavior fires is not explicitly documented in available sources — treat as unconfirmed.

Predefined gateways (gateway mode, hardware devices only)

LAN CTR (LAN Customer Trusted Resolver) and WAN CTR are pre-created, non-editable gateways for hardware Branch Connector devices in gateway mode. They are disabled by default and have no effect in Cloud Connector virtual-appliance deployments.

DNS Policies

DNS Policies define rule-based control over DNS requests and responses. A key architectural distinction: DNS Policies inspect all DNS traffic regardless of transport, including UDP, TCP, and DNS-over-HTTPS (DoH). This is broader than ZIA DNS Control, which only sees plaintext port-53 traffic unless DoH flows are separately decrypted.

Rule evaluation

Rules evaluate in ascending numerical order; first match wins. Two predefined rules ship:

Redirect Resolution of Zscaler Domains to WAN CTR
Disabled by default. Matches the Zscaler Cloud Endpoints app service group. Applies only to hardware gateway-mode devices.
Default Connector DNS Rule
Enabled (allow all). Catch-all; always lowest precedence. Action is modifiable but the rule cannot be deleted. New filtering rules must be placed above it to have any effect.

DNS Policy actions

ActionEffect
AllowPermit the DNS transaction.
BlockDeny the DNS transaction.
Resolved by ZPAHand off resolution to ZPA — used for ZPA split-DNS where private-app domains resolve to ZPA synthetic IPs.
RedirectRoute the query to a specific DNS server / DNS Gateway.
Overwrite DNS responseRewrite the response returned to the client — for sinkholing, steering internal FQDNs, or ZPA split-DNS.

DoH and DNS tunnel detection

DNS Policies inspect DoH at the application layer, identifying encrypted DNS queries without requiring separate SSL inspection of the HTTPS flow. DNS tunnel detection is a stated capability; exact trigger criteria and response actions are not in available source captures.

Interaction with Log & Control Forwarding

DNS Policy outcomes are logged via Log & Control Forwarding. A block or redirect with no visible cloud event usually means Log & Control Forwarding is broken, not that the policy rule did not fire. Log gaps and rule misfires appear identical from the cloud-side view.

Conversely, a Log & Control Forwarding outage can cause CC to run stale DNS Policy rules or stale DNS Gateway definitions — the CC fails to receive policy updates. When diagnosing stale policy behavior, test Log & Control Forwarding connectivity separately from workload-path testing.

§ 5 · Credential & upgrade lifecycle

Credential & upgrade lifecycle

Weekly software upgrades are Zscaler-triggered and staggered within a group. OS image upgrades require a re-deploy cycle. zsroot credential rotation is per-device, operator-initiated.

Software upgrade cadence

DimensionDetail
Check frequencyOnce per week.
Default windowSunday at midnight, local time of the cloud region where the VM runs — not operator timezone, not UTC unless the region happens to be UTC.
Within-group behaviorStaggered across the 2-hour window; at most one CC per group is down at a time. Requires ≥2 healthy members before the window opens.
Failed upgradesRetry at the next weekly window. Failed upgrades do not block the rest of the group.
Status fieldScheduled / Success / Failure per CC in Admin Console → Infrastructure → Connectors → Cloud.
2-CC group has zero redundancy during upgrades A group with exactly 2 CCs passes the weekly window with zero redundancy while the first CC restarts. Zscaler's documented production minimum is 2 CCs per AZ across 2 AZs (4 total). A 2-CC deployment is below the documented redundancy minimum.

Software package vs OS image

ComponentHow it upgrades
Zscaler software packageAutomatic, weekly, in-place. No re-provisioning required.
OS image (VM image)Not in-place. Requires creating a new deployment with the updated marketplace image. OS-level CVEs require a re-deploy cycle, not just waiting for the next Sunday window.

Configuring the upgrade window

Navigate: Admin Console → Infrastructure → Connectors → Cloud → [select group] → Edit. The upgrade schedule field accepts a day-of-week and time. The schedule is group-scoped — no per-CC override. Shift groups away from Sunday midnight default to avoid overlap with tenant maintenance blackouts.

VMSS and OS upgrades

For Azure VMSS deployments, a new VMSS instance launched from a fresh image already has current software — the VMSS scale-out event functions as a de-facto OS upgrade. Treat VMSS re-deployments with updated images as OS-level upgrades rather than waiting for the weekly package window.

zsroot credential rotation

Every CC and Branch Connector VM has a privileged local service account named zsroot, used for administrative SSH access and internal device management. It is not the account used for control-plane registration or cloud-provider API calls.

Access path by platform

PlatformNormal login methodRotation procedure
Cloud Connector (public cloud VM)Cloud-provider SSH key; zsroot password not required for SSH in this modelssh <cc-host> (using cloud-provider key), then passwd zsroot
Branch Connector (virtual)Console/management interface SSH to 169.254.2.2ssh zsroot@169.254.2.2, then passwd zsroot

Rotation is per-device — there is no tenant-wide or group-wide rotation API. Each CC VM must be rotated individually. Zscaler mandates no specific interval; align to the tenant's compliance framework.

Re-deploy resets the password

When a CC is re-provisioned to pick up an OS image upgrade, the zsroot password is reset with the new VM. Rotation records should note the re-deploy date; the rotation clock restarts. Rotate after OS image re-deploys, not before.

Recommended sequencing

For in-place package upgrades: rotate before the upgrade window (closes the "stale credential during upgrade" gap). For OS image re-deploys: plan the re-deploy first, then rotate zsroot on new VMs post-provisioning before the group goes into production.
